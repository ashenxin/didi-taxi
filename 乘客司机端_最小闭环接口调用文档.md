# 乘客/司机端最小闭环：接口调用文档（MVP）

本文档描述 **乘客下单 → 系统派单（最近司机）→ 司机接单 → 行程推进 → 完单计费** 的端到端接口调用路线，目标是让 `didi-passenger-h5` / `didi-driver-h5` 可以按本文档完成联调。

适用范围：
- 乘客端 H5：`didi-passenger-h5`（经网关调用 `passenger-api`）
- 司机端 H5：`didi-driver-h5`（经网关调用 `driver-api`）
- 聚合层：`passenger-api(8100)`、`driver-api(8101)`
- 核心服务：`order(8093)`、`capacity(8090)`、`calculate(8091)`
- 地图服务：`map-service(8094)` — **已接入高德 Web 服务 Demo**；H5 **经网关只访问 BFF**，地理编码与驾车规划由 **`passenger-api` 内 Feign 调 map**，浏览器不直连 map（排障除外）。
- **网关**：`gateway(8080)`；校验 `Authorization: Bearer` 后注入 **`X-User-Id`**（值为 JWT `sub`）。客户端**不要**、也**不要尝试**伪造 `X-User-Id`（网关会剥离后再注入）。

---

## 0. 网关、JWT 与 `X-User-Id`（与当前实现对齐）

### 0.1 浏览器 / H5 该怎么传

| 头 / 行为 | 说明 |
|-----------|------|
| **`Authorization`** | 必填（除登录、发码等**白名单**接口）。格式：`Bearer <accessToken>`。乘客端、司机端各自登录接口返回的 `accessToken`。 |
| **`X-User-Id`** | **不要由前端设置**。由网关在验签通过后写入，下游 BFF 从该头读取当前用户 ID（乘客 `customerId` / 司机 `driverId`）。 |
| **直连 BFF**（`8100` / `8101`） | 无网关时不会自动带 `X-User-Id`，需 Postman 等**手动加** `X-User-Id` 与合法 JWT 场景一致；**联调 H5 请统一走 `http://127.0.0.1:8080`**。 |

### 0.2 网关白名单（无需 Bearer）

- 乘客：`POST /app/api/v1/auth/sms/send`、`login-sms`、`login-password`
- 司机：`POST /driver/api/v1/auth/sms/send`、`register-sms`、`register-password`、`login-sms`、`login-password`
- 管理端：`POST /admin/api/v1/auth/login`（与乘客/司机无关）
- `GET /actuator/health`（健康检查）

其余 `/app/**`、`/driver/**` 在 **`gateway.jwt.require-auth=true`**（默认）下**必须**带合法 JWT；网关按路径选用密钥并校验 **`aud`**（乘客 `app-bff`，司机 `driver-bff`，与各自 BFF 签发配置一致）。

### 0.3 `GATEWAY_JWT_REQUIRE_AUTH=false`（仅本地排障）

关闭强制鉴权时，网关**仍会在请求带合法 Bearer 时**解析 JWT 并注入 `X-User-Id`；若无 Token 或 Token 无效，请求原样转发，BFF 因缺少 `X-User-Id` 返回 **401「未授权，请重新登录」**。因此即使用该开关，**下单等业务仍须带登录拿到的 Token**。

## 1. 关键概念澄清

### 1.1 抢单 vs 派单

- **抢单（broadcast）**：订单发布给一定范围内的多个司机，司机端列表里可见，谁先点击接单谁拿到（需要并发控制与 offer 管理）。
- **派单/指派（dispatch/assign）**：系统选择一个司机（通常是“最近且可接单”），将订单 **指派** 给他；司机端只看到分配给自己的订单。

你当前 MVP 选择：**派单给最近司机**（不做广播抢单）。

### 1.2 轮询 vs 长连接

现实中打车软件通常使用长连接（WebSocket/自研长链）推送新订单、状态变化、位置更新。

第一版（MVP）采用：
- **轮询（polling）**：实现最快、无需额外基础设施。

建议轮询频率（可按实际压测调整）：
- **乘客端**：`GET /app/api/v1/orders/{orderNo}` 每约 2 秒，请求带 **`Authorization`**（网关注入 **`X-User-Id`**）
- **司机端**：`GET /driver/api/v1/orders/assigned` 每 1–2 秒，请求带 **`Authorization`**；Query **`driverId` 可省略**（省略时默认当前登录司机）；若传了则**必须与 JWT 身份一致**，否则 **403**

后续可选升级（按需要再做）：
- **长轮询（long-polling）**：更省流量
- **SSE**：单向推送，适合状态通知
- **WebSocket**：体验最好，实现最复杂

落地顺序：**先轮询跑通闭环** → 再升级到 **SSE/WS**。

### 1.3 地图（高德）与调用分层（与当前 `passenger-api` 一致）

**下单闭环（B 层）**已由 **`passenger-api` 内 `MapClient`** 直连 **map-service**：

1. **地理编码（按需）**：若 `origin`/`dest` **未同时提供 `lat`+`lng`**，则用 `address` 或 `name` 拼地址，调用 **`GET /api/v1/map/demo/amap-geocode`** 补全经纬度。BFF 内对 `cityCode` 有**有限映射**（例如 `330100` → 高德 `city=杭州`），其它城市需后续扩展映射表或改传坐标。
2. **驾车规划**：起终点坐标齐备后调用 **`POST /api/v1/map/demo/amap-driving`**（`RouteRequest`），得到 `distanceMeters` / `durationSeconds`，再进入 **`calculate.estimate`** → 建单 → 派单。

| 层级 | 谁调用 | 作用 | 现状 |
|------|--------|------|------|
| **A. H5 选点 / 地图 UI（可选）** | 浏览器（如高德 JS SDK） | 选点、展示、直接拿 lat/lng | 可将坐标传给 BFF，**跳过**服务端 geocode |
| **B. 一步下单（必走 BFF）** | **passenger-api** Feign → map | geocode（缺坐标时）+ 驾车规划 + 预估 | **已接高德 Demo 路径**，非 stub `/map/route` |

H5 **不必**为闭环再调地图 HTTP：只要 BFF 能 geocode（中文地址 + 已支持的城市）或前端传齐经纬度即可。

### 1.4 ETA 是什么（防忘）

- **ETA** = **Estimated Time of Arrival**，中文常称 **预计到达时间**。
- 在本打车场景里，若无特别说明，一般指：**被指派/接单后的司机，从当前位置开到乘客「上车点」预计还要多久**（不是乘客到终点的全程时间，除非接口文档单独写明）。
- 文档或 JSON 里的 **`etaSeconds`**：表示上述 ETA 的 **秒数**（例如 `260` ≈ 4 分 20 秒后到达上车点）。

## 2. 订单状态机（最小）

### 2.1 状态枚举（语义）

- `CREATED`：已创建（下单成功）
- `ASSIGNED`：已派单（已指派司机，待司机确认）
- `ACCEPTED`：司机已接单
- `ARRIVED`：司机已到达上车点
- `STARTED`：行程开始
- `FINISHED`：行程结束（已结算价）
- `CANCELLED`：已取消

### 2.2 允许的流转（MVP）

- `CREATED → ASSIGNED → ACCEPTED → ARRIVED → STARTED → FINISHED`
- `CREATED / ASSIGNED / ACCEPTED` 可 `CANCELLED`（规则后续再细化）

### 2.3 并发与幂等（必须）

所有写操作建议满足：
- **CAS 更新**：`where order_no=? and status=?`（或带 version 字段）
- **幂等**：同一操作重复调用要么返回成功（已是目标状态），要么返回冲突码
- **防串单**：司机侧写操作需校验 `driverId == order.driverId`

## 3. 对外 API（乘客端）—— passenger-api

统一前缀：`/app/api/v1`（经网关的完整 URL 形如 `http://127.0.0.1:8080/app/api/v1/...`）

### 3.0 【特别注明】推荐落地方案（避免分布式事务）

为兼顾后续系统一致性与安全性，本文档采用如下约定（**推荐**）：
- **对外接口形态：一步**（下单一次调用即可）
  - 乘客端只调用一次下单接口，即可拿到 `orderNo`，以及“是否已指派司机”的结果
- **对内实现：两段式**（避免分布式事务）
  - **第一段（强一致，本地事务）**：创建订单为 `CREATED` + 写 `order_event(ORDER_CREATED)`（或 outbox 事件）
  - **第二段（最终一致，可重试）**：找司机并指派，成功则 `ASSIGNED`；失败/超时则保持 `CREATED`，由调度器/重试任务继续处理

收益：
- 不引入 2PC/TCC 等分布式事务复杂度
- 更容易做幂等、重试与故障恢复
- 对前端更简单（配合轮询即可看到状态推进）

第一版实现建议：
- 可以先做同步版 `createAndAssign`（同一请求内顺序调用 `capacity` 并 `assign`），但要确保：
  - 幂等（如支持 `Idempotency-Key`）
  - CAS 更新（状态机）
  - 超时降级（找不到司机就返回 `CREATED`，后续由重试/调度器补指派）

当前仓库实现：**同步版** `createAndAssign` 已落地（见 `passenger-api` `PassengerOrderService`），与上表「两段式异步指派」可后续再演进。

### 3.0a 乘客端 H5：推荐调用顺序

前提：**经网关 `8080`**，业务请求带 **`Authorization: Bearer <accessToken>`**（见 **§0**）。

1. **发码 / 登录（白名单，可不带 Token）**  
   - `POST /app/api/v1/auth/sms/send` — body：`{ "phone": "13800138000" }`  
   - `POST /app/api/v1/auth/login-sms` — `{ "phone", "code" }`  
   - 或 `POST /app/api/v1/auth/login-password` — `{ "phone", "password" }`  
   响应 `data` 中含 **`accessToken`**、`tokenType`（`Bearer`）、`expiresIn`、`customer`（含 `id` 等）。前端保存 Token，后续请求写入 `Authorization`。
2. **（可选，产品层）选点**  
   H5 用地图 SDK 取 **lat/lng** 可直接写入下单 body；仅中文地址时依赖 BFF **geocode**（见 §1.3 城市映射限制）。
3. **一步下单（需登录）**  
   `POST /app/api/v1/orders` — 仅带 **`Authorization`** 即可；**不要**传 `X-User-Id`。  
   Body 中 **`passengerId` 可省略**；若填写则**必须与** JWT 对应用户一致，否则 **400**（「请求体中的乘客信息与当前登录身份不一致…」）。服务端最终以 **`X-User-Id`（网关注入）为准** 写入订单。
4. **轮询订单**  
   `GET /app/api/v1/orders/{orderNo}` — 带 **`Authorization`**。
5. **（可选）取消**  
   `POST /app/api/v1/orders/{orderNo}/cancel` — body 可只含 `cancelReason`；`passengerId` 同样由服务端从 `X-User-Id` 注入，规则同下单。

### 3.1 下单（创建 + 派单 + 返回预估）

**POST** `/app/api/v1/orders`（经网关即路径不变）

**请求头（浏览器）**：`Authorization: Bearer <accessToken>`。`X-User-Id` 由网关注入，**勿在浏览器手动设置**。

**请求体校验要点**：

- `provinceCode`、`cityCode`、`productCode`：必填字符串。
- `origin`、`dest`：必填；子字段 **`name` 必填**；`address` 可选。  
  - 若 **`lat`、`lng` 均缺**：须能通过 `address` 或 `name` 拼出可 geocode 的地址（见 §1.3）。  
  - 若 **`lat`、`lng` 均有**：跳过 geocode，直接驾车规划。

**示例 A：仅中文地址（H5 Demo 常见）**

```jsonc
{
  "provinceCode": "330000",
  "cityCode": "330100",
  "productCode": "ECONOMY",
  "origin": {
    "name": "杭州火车东站",
    "address": "浙江省杭州市上城区全福桥路2号杭州东站"
  },
  "dest": {
    "name": "龙翔桥地铁站",
    "address": "浙江省杭州市上城区湖滨街道龙翔桥地铁站"
  }
}
```

**示例 B：已选点坐标 + 名称**

```jsonc
{
  "provinceCode": "330000",
  "cityCode": "330100",
  "productCode": "ECONOMY",
  "origin": { "lat": 30.251612, "lng": 120.141275, "name": "上车点", "address": "可选" },
  "dest": { "lat": 30.292753, "lng": 120.21201, "name": "下车点", "address": "可选" }
}
```

**响应 `data` 形态（与 `CreateAndAssignOrderResult` 一致）**：

```jsonc
{
  "code": 200,
  "msg": "success",
  "data": {
    "orderNo": "20260330xxxx",
    "status": { "code": 1, "en": "ASSIGNED", "zh": "已派单" },
    "route": {
      "distanceMeters": 12345,
      "durationSeconds": 1560,
      "provider": "amap",
      "traceId": "..."
    },
    "estimate": {
      "ruleId": 1,
      "estimatedAmount": 28.5,
      "distanceMeters": 12345,
      "durationSeconds": 1560
    },
    "assignedDriver": {
      "driverId": 80001,
      "carId": 1,
      "companyId": 1,
      "carNo": "浙A10001",
      "etaSeconds": 260
    }
  }
}
```

说明：`assignedDriver` 在无可用司机时可为 **null**，此时 `status` 多为 **`CREATED`**。

**passenger-api 对内调用链（与代码一致，按序）：**

1. **（按需）`map-service`** `GET /api/v1/map/demo/amap-geocode` — 补全缺省坐标的起/终点。  
2. **`map-service`** `POST /api/v1/map/demo/amap-driving` — 驾车里程/时长。  
3. **`calculate-service`** `POST /api/v1/calculate/estimate`。  
4. **`order-service`** `POST /api/v1/orders` — 创建订单。  
5. **`capacity-service`** `GET /api/v1/dispatch/nearest-driver`。  
6. **`order-service`** `POST /api/v1/orders/{orderNo}/assign` — 有司机则 `ASSIGNED`。

备注：对外仍 **一次** `POST /orders`；对内异步重试指派见 §3.0，与当前同步实现可并行演进。

### 3.3 取消订单

**POST** `/app/api/v1/orders/{orderNo}/cancel`

**请求头**：`Authorization: Bearer <token>`。

请求体（`passengerId` 可省略；若传须与 JWT 身份一致，否则 **400**）：

```jsonc
{
  "cancelReason": "不需要了"
}
```

服务内调用：`order.cancel`（CAS：只允许从 `CREATED/ASSIGNED/ACCEPTED` 取消）。

## 4. 对外 API（司机端）—— driver-api

统一前缀：`/driver/api/v1`（经网关完整 URL 形如 `http://127.0.0.1:8080/driver/api/v1/...`；网关转发至 `driver-api:8101`，路径不变）

**鉴权**：业务接口须带 **`Authorization: Bearer`**；网关注入 **`X-User-Id`**（字符串形式的 `driverId`）。路径参数、Query、JSON 里的 **`driverId` 若出现，必须与 `X-User-Id` 一致**，否则 **403「禁止操作其他司机数据」**。

### 4.0 司机端认证（白名单，无需 Token）

| 接口 | 说明 |
|------|------|
| `POST /driver/api/v1/auth/sms/send` | 发码；body：`{ "phone" }` |
| `POST /driver/api/v1/auth/register-sms` | 短信注册并登录；`{ "phone", "code" }` |
| `POST /driver/api/v1/auth/register-password` | 短信验证后设密并登录；`{ "phone", "code", "password" }` |
| `POST /driver/api/v1/auth/login-sms` | 验证码登录 |
| `POST /driver/api/v1/auth/login-password` | 密码登录 |

响应 `data` 中含 **`accessToken`** 等（与乘客端结构类似，字段以 `DriverLoginResponse` 为准）。

**推荐调用顺序（闭环）**：任选注册/登录拿 Token → `POST .../drivers/{driverId}/online` → 轮询 `GET .../orders/assigned` → `accept` → `arrive` → `start` → `finish`。

### 4.1 司机上线/下线（影响派单候选集）

**POST** `/driver/api/v1/drivers/{driverId}/online`

**请求头**：`Authorization: Bearer <token>`；`{driverId}` 须与 JWT / `X-User-Id` 一致。

请求体：

```jsonc
{
  "online": true // 是否上线（true 上线可派单，false 下线不参与派单）
}
```

服务内调用：`capacity` 更新 `monitor_status`（0/1/2 口径按你现有字段）。

### 4.2 拉取“分配给我的订单”（派单模式）

**GET** `/driver/api/v1/orders/assigned`  
可选 Query：**`driverId`** — 省略时默认使用 **`X-User-Id`** 对应司机；传入时须与登录身份一致。

**请求头**：`Authorization: Bearer <token>`。

响应（示例）：

```jsonc
{
  "code": 200, // 业务码
  "msg": "success", // 提示信息
  "data": [
    {
      "orderNo": "20260330xxxx", // 订单号
      "status": "ASSIGNED", // 订单状态（通常是已指派 ASSIGNED）
      "pickup": {
        "name": "西湖景区" // 上车点名称（可扩展经纬度/地址）
      },
      "etaSeconds": 260 // 预计到达上车点时间（秒）
    }
  ]
}
```

服务内调用：`order.listAssignedToDriver`（按 `driver_id` + `status in (...)` 查询）。

### 4.3 接单（确认指派）

**POST** `/driver/api/v1/orders/{orderNo}/accept`

**请求头**：`Authorization: Bearer <token>`。

请求体（`driverId` 须等于当前登录司机）：

```jsonc
{
  "driverId": 80001
}
```

服务内调用顺序（建议）：
1) `capacity.checkDriverCanAccept`：`can_accept_order=1` 且在线
2) `order.accept`：CAS（只允许 `ASSIGNED → ACCEPTED`，且 `driver_id` 必须匹配指派司机）

### 4.4 到达 / 开始 / 完单

均需 **`Authorization`**；body 中 **`driverId` 与登录身份一致**（`finish` 成功后会以服务端为准写回 `driverId`）。

- **POST** `/driver/api/v1/orders/{orderNo}/arrive` — body：`{ "driverId" }`；`ACCEPTED → ARRIVED`
- **POST** `/driver/api/v1/orders/{orderNo}/start` — body：`{ "driverId" }`；`ARRIVED → STARTED`
- **POST** `/driver/api/v1/orders/{orderNo}/finish` — body（MVP）：

```jsonc
{
  "driverId": 80001,
  "distanceKm": 12.3,
  "durationMin": 26,
  "finalAmount": null
}
```

`finalAmount` 可选；`distanceKm` / `durationMin` 可选，按订单服务与计费演进使用。

服务内调用顺序（建议）：
1) `calculate.settle`：计算 `finalAmount`（基于 fare_rule + metrics）
2) `order.finish`：`STARTED → FINISHED` + 写入 `final_amount` + 写事件

## 5. 核心服务对内 API（建议在 order-service 提供）

> 这里列的是“被 BFF 调用”的核心接口形态。你已有部分订单查询/事件能力，可在此基础上补齐写接口。

统一前缀（示例）：`/api/v1`

### 5.1 创建订单

## 6. 轮询与推送：推荐实现方式

### 6.1 MVP 轮询建议（最小闭环）

- 乘客端：`GET /app/api/v1/orders/{orderNo}` 每约 2s，带 **`Authorization`**
- 司机端：`GET /driver/api/v1/orders/assigned` 每 1–2s，带 **`Authorization`**（`driverId` Query 可省略）

### 6.2 升级方案

**方案 1：长轮询**
- `GET /.../watch?sinceVersion=...&timeout=25s`
- 有变化立即返回；否则超时返回空

**方案 2：SSE**
- `GET /.../events/stream`
- 前端 `EventSource` 订阅订单状态变更

**方案 3：WebSocket**
- 单连接推送：新单/状态/位置
- 体验最好，实现复杂度最高

## 7. 错误码与返回结构（建议约定）

统一返回：`ResponseVo`（`code` / `msg` / `data`）；成功 **`code === 200`**。

重点错误：
- **400**：参数校验失败；乘客端 body **`passengerId` 与登录身份不一致**
- **401**：未带 Token、Token 无效/过期；或 BFF **未收到 `X-User-Id`**（常见：未走网关、或关鉴权但未带 Bearer）
- **403**：乘客看他人订单；司机 **`driverId` 与 JWT 不一致**
- **404**：订单不存在
- **409**：状态冲突（例如非 `ASSIGNED` 仍尝试 `accept`）

## 9. 自测记录与「仍为模拟 / 待实现」说明（便于后续迭代）

> 本节用于记录：**当前仓库已能跑通的最小闭环**，以及**仍为占位、假数据或与真实业务有差距**的部分，避免联调时误以为已是生产级能力。

### 9.1 已通过自测（本地联调）

在 **MySQL 各库与 seed 数据就绪**（如 `passenger.customer`、`calculate.fare_rule`、`capacity` 派单演示司机/车辆等）且 **网关 + 相关服务均已启动** 的前提下，已自测可走通：

- **乘客端**：登录拿 Token → 经 **`8080`** `POST /app/api/v1/orders`（`Authorization`，body 可不传 `passengerId`）→ 返回 `orderNo`；有可派单司机时 **`ASSIGNED`**；`GET /app/api/v1/orders/{orderNo}` 轮询状态推进。
- **司机端**：登录 → `POST /driver/api/v1/drivers/{driverId}/online` → `GET /driver/api/v1/orders/assigned` → `accept` / `arrive` / `start` / `finish` 串联后，订单状态可到 **已完成（FINISHED）**。
- **订单服务**：状态机写操作与事件流水与当前实现一致（详见 `order-service` 代码与库表）。

> **说明**：自测依赖本地/测试环境数据；若派不到单（无满足条件的司机+车），订单会停留在 **CREATED**，属预期，需补 `capacity` 数据或检查司机是否已上线听单。

### 9.2 仍为模拟、简化或与文档「理想形态」不一致的实现（待你后续补齐）

| 领域 | 当前行为 | 后续建议 |
|------|----------|----------|
| **地图 `map-service` + BFF** | `passenger-api` 已 Feign：**geocode**（`GET .../amap-geocode`）+ **驾车**（`POST .../amap-driving`） | 扩展 **`cityCode → geocode city`** 映射；必要时 **matrix** 做真实司机 ETA |
| **派单 `capacity`「最近司机」** | `nearest-driver` 为 **按城市+车型等条件筛选后的首条候选**，**不是**真实地理距离最近 | 引入司机实时位置、距离排序、并发与拒单策略等 |
| **下单返回里的 ETA** | `assignedDriver.etaSeconds` 等与 **派单逻辑**相关；**全程路线时长**来自 **高德驾车规划**，**不是**「司机当前位置 → 上车点」的接驾 ETA | 接地图 matrix + 司机实时坐标 |
| **计费 `calculate`** | 已有 **`estimate`**（预估）；**完单链路未调用 `settle`** | 实现 **`settle`/后结算** 并在 `finish` 前或内写入真实 `final_amount` 规则 |
| **订单 `finish` 金额** | 若未传实付，`order-service` 可用 **`estimated_amount` 兜底** 写入 `final_amount` | 与 `calculate.settle`、优惠/抽成等对齐后再落库 |
| **司机端指派列表 VO** | `GET /driver/api/v1/orders/assigned` 返回中 **`etaSeconds` 等可为 null**（MVP 未算） | 派单或列表接口补充真实 ETA |
| **乘客下单 BFF** | 同步编排（map → estimate → create → dispatch → assign），**非**文档 §3.0 理想的「对内两段式 + 异步重试指派」 | 按需引入调度/重试、幂等键、超时降级 |
| **分布式追踪** | `pom` 中 Zipkin/Brave 依赖已注释；`application.yml` 默认 **`export.zipkin.enabled: false`** | 需要链路时再恢复依赖、起 Zipkin 并打开开关 |
| **长连接 / 推送** | 文档中的 SSE/WebSocket **未实现**；当前靠 **轮询** | 按优先级选型实现 |
| **司机换队等 App 能力** | `capacity` 侧存在 **501 占位**接口（若仍保留） | 按业务文档替换为真实提交/审核流 |

### 9.3 联调时建议固定的最小数据约定（与 seed 脚本对齐）

- 乘客：登录用户 JWT `sub` / 网关注入的 `X-User-Id` 与库表 **`customer.id`** 一致且 `is_deleted=0`（body 可不传 `passengerId`）。
- 城市/产品线：与 `fare_rule` 的 `province_code` / `city_code` / `product_code`（如 `ECONOMY`）一致，否则 **estimate** 会失败。
- 派单：`driver` 的 `can_accept_order`、`monitor_status`、与 `car` 的 `city_code`、`ride_type_id`、`car_state` 等满足当前 `nearest-driver` 筛选逻辑。
