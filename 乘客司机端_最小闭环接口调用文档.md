# 乘客/司机端最小闭环：接口调用文档（MVP）

本文档描述 **乘客下单 → 系统派单（最近司机）→ 司机接单 → 行程推进 → 完单计费** 的端到端接口调用路线，目标是让 `didi-passenger-h5` / `didi-driver-h5` 可以按本文档完成联调。

适用范围：
- 乘客端 H5：`didi-passenger-h5`（调用 `passenger-api`）
- 司机端 H5：`didi-driver-h5`（调用 `driver-api`）
- 聚合层：`passenger-api(8100)`、`driver-api(8101)`
- 核心服务：`order(8093)`、`capacity(8090)`、`calculate(8091)`
- 地图服务：`map`（后续接入，可先 stub）

---

## 1. 关键概念澄清

### 1.1 抢单 vs 派单

- **抢单（broadcast）**：订单发布给一定范围内的多个司机，司机端列表里可见，谁先点击接单谁拿到（需要并发控制与 offer 管理）。
- **派单/指派（dispatch/assign）**：系统选择一个司机（通常是“最近且可接单”），将订单 **指派** 给他；司机端只看到分配给自己的订单。

你当前 MVP 选择：**派单给最近司机**（不做广播抢单）。

### 1.2 轮询 vs 长连接

现实中打车软件通常使用长连接（WebSocket/自研长链）推送新订单、状态变化、位置更新。

第一版（MVP）采用：
- **轮询（polling）**：实现最快、无需额外基础设施。

建议轮询频率（可按实际压测调整）：
- **乘客端**：`GET /app/api/v1/orders/{orderNo}` 每 2 秒
- **司机端**：`GET /driver/api/v1/orders/assigned?driverId=...` 每 1–2 秒

后续可选升级（按需要再做）：
- **长轮询（long-polling）**：更省流量
- **SSE**：单向推送，适合状态通知
- **WebSocket**：体验最好，实现最复杂

落地顺序：**先轮询跑通闭环** → 再升级到 **SSE/WS**。

### 1.3 ETA 是什么（防忘）

- **ETA** = **Estimated Time of Arrival**，中文常称 **预计到达时间**。
- 在本打车场景里，若无特别说明，一般指：**被指派/接单后的司机，从当前位置开到乘客「上车点」预计还要多久**（不是乘客到终点的全程时间，除非接口文档单独写明）。
- 文档或 JSON 里的 **`etaSeconds`**：表示上述 ETA 的 **秒数**（例如 `260` ≈ 4 分 20 秒后到达上车点）。

## 2. 订单状态机（最小）

### 2.1 状态枚举（语义）

- `CREATED`：已创建（下单成功）
- `ASSIGNED`：已派单（已指派司机，待司机确认）
- `ACCEPTED`：司机已接单
- `ARRIVED`：司机已到达上车点
- `STARTED`：行程开始
- `FINISHED`：行程结束（已结算价）
- `CANCELLED`：已取消

### 2.2 允许的流转（MVP）

- `CREATED → ASSIGNED → ACCEPTED → ARRIVED → STARTED → FINISHED`
- `CREATED / ASSIGNED / ACCEPTED` 可 `CANCELLED`（规则后续再细化）

### 2.3 并发与幂等（必须）

所有写操作建议满足：
- **CAS 更新**：`where order_no=? and status=?`（或带 version 字段）
- **幂等**：同一操作重复调用要么返回成功（已是目标状态），要么返回冲突码
- **防串单**：司机侧写操作需校验 `driverId == order.driverId`

## 3. 对外 API（乘客端）—— passenger-api

统一前缀：`/app/api/v1`

### 3.0 【特别注明】推荐落地方案（避免分布式事务）

为兼顾后续系统一致性与安全性，本文档采用如下约定（**推荐**）：
- **对外接口形态：一步**（下单一次调用即可）
  - 乘客端只调用一次下单接口，即可拿到 `orderNo`，以及“是否已指派司机”的结果
- **对内实现：两段式**（避免分布式事务）
  - **第一段（强一致，本地事务）**：创建订单为 `CREATED` + 写 `order_event(ORDER_CREATED)`（或 outbox 事件）
  - **第二段（最终一致，可重试）**：找司机并指派，成功则 `ASSIGNED`；失败/超时则保持 `CREATED`，由调度器/重试任务继续处理

收益：
- 不引入 2PC/TCC 等分布式事务复杂度
- 更容易做幂等、重试与故障恢复
- 对前端更简单（配合轮询即可看到状态推进）

第一版实现建议：
- 可以先做同步版 `createAndAssign`（同一请求内顺序调用 `capacity` 并 `assign`），但要确保：
  - 幂等（如支持 `Idempotency-Key`）
  - CAS 更新（状态机）
  - 超时降级（找不到司机就返回 `CREATED`，后续由重试/调度器补指派）

### 3.1 下单（创建 + 派单 + 返回预估）

**POST** `/orders`

请求体（MVP 示例）：

```jsonc
{
  "passengerId": 1, // 乘客 ID（MVP 可先用 mock）
  "provinceCode": "330000", // 省份编码（如浙江 330000）
  "cityCode": "330100", // 城市编码（如杭州 330100）
  "productCode": "ECONOMY", // 产品线/车型（如 ECONOMY 经济型）
  "origin": {
    "lat": 30.251612, // 起点纬度
    "lng": 120.141275, // 起点经度
    "name": "西湖景区", // 起点 POI 名称
    "address": "..." // 起点详细地址（可选）
  },
  "dest": {
    "lat": 30.292753, // 终点纬度
    "lng": 120.21201, // 终点经度
    "name": "杭州东站", // 终点 POI 名称
    "address": "..." // 终点详细地址（可选）
  }
}
```

响应（建议）：

```jsonc
{
  "code": 200, // 业务码（示例用 200 表示成功，实际以 ResponseVo 约定为准）
  "msg": "success", // 提示信息
  "data": {
    "orderNo": "20260330xxxx", // 订单号
    "status": "ASSIGNED", // 订单当前状态（已指派）
    "estimated": {
      "distanceMeters": 12345, // 预估里程（米）
      "durationSeconds": 1560, // 预估时长（秒）
      "estimatedAmount": 28.5 // 预估费用（元）
    },
    "assignedDriver": {
      "driverId": 80001, // 司机 ID
      "driverName": "张师傅", // 司机姓名/称呼
      "carNo": "浙A10001", // 车牌号（或车辆编号）
      "etaSeconds": 260 // 司机到达上车点 ETA（秒）
    }
  }
}
```

服务内调用链（建议顺序）：
1) `map.route`：起终点路线距离/时长（MVP 可先 stub 固定值）
2) `calculate.estimate`：预估价（可先简化规则）
3) `order.create`：创建订单（状态 `CREATED`）+ 写 `order_event(ORDER_CREATED)`（或 outbox）
4) `capacity.searchNearestDriver`：找“最近且在线且可接单”的司机（MVP 可先按 city 取一个）
5) `order.assign`：指派司机（状态 `ASSIGNED`，落库 driverId/carId/companyId，可选 etaSeconds）+ 写 `order_event(ORDER_ASSIGNED)`

备注：
- 为减少 BFF 编排与降低一致性风险，推荐让 `order-service` 对外提供 **`createAndAssign`**（下单只调一次）。

### 3.2 查询订单详情（用于轮询）

**GET** `/orders/{orderNo}`

响应（示例）：

```jsonc
{
  "code": 200, // 业务码
  "msg": "success", // 提示信息
  "data": {
    "orderNo": "20260330xxxx", // 订单号
    "status": "ACCEPTED", // 订单当前状态（已接单）
    "estimatedAmount": 28.5, // 预估费用（元）
    "finalAmount": null, // 最终费用（元，完单后才有；未完单为 null）
    "driver": {
      "driverId": 80001, // 司机 ID
      "driverName": "张师傅", // 司机姓名/称呼
      "carNo": "浙A10001" // 车牌号
    },
    "timestamps": {
      "createdAt": "2026-03-30 10:00:00", // 下单时间
      "acceptedAt": "2026-03-30 10:00:10" // 接单时间
    }
  }
}
```

服务内调用：`order.detail`（必要时再聚合 driver/car 信息）。

### 3.3 取消订单

**POST** `/orders/{orderNo}/cancel`

请求体：

```jsonc
{
  "cancelReason": "不需要了" // 取消原因（用于乘客侧备注/运营分析）
}
```

服务内调用：`order.cancel`（CAS：只允许从 `CREATED/ASSIGNED/ACCEPTED` 取消）。

## 4. 对外 API（司机端）—— driver-api

统一前缀：`/driver/api/v1`

### 4.1 司机上线/下线（影响派单候选集）

**POST** `/drivers/{driverId}/online`

请求体：

```jsonc
{
  "online": true // 是否上线（true 上线可派单，false 下线不参与派单）
}
```

服务内调用：`capacity` 更新 `monitor_status`（0/1/2 口径按你现有字段）。

### 4.2 拉取“分配给我的订单”（派单模式）

**GET** `/orders/assigned?driverId=80001`

响应（示例）：

```jsonc
{
  "code": 200, // 业务码
  "msg": "success", // 提示信息
  "data": [
    {
      "orderNo": "20260330xxxx", // 订单号
      "status": "ASSIGNED", // 订单状态（通常是已指派 ASSIGNED）
      "pickup": {
        "name": "西湖景区" // 上车点名称（可扩展经纬度/地址）
      },
      "etaSeconds": 260 // 预计到达上车点时间（秒）
    }
  ]
}
```

服务内调用：`order.listAssignedToDriver`（按 `driver_id` + `status in (...)` 查询）。

### 4.3 接单（确认指派）

**POST** `/orders/{orderNo}/accept`

请求体：

```jsonc
{
  "driverId": 80001 // 当前操作的司机 ID（用于校验“是否是被指派的司机”）
}
```

服务内调用顺序（建议）：
1) `capacity.checkDriverCanAccept`：`can_accept_order=1` 且在线
2) `order.accept`：CAS（只允许 `ASSIGNED → ACCEPTED`，且 `driver_id` 必须匹配指派司机）

### 4.4 到达 / 开始 / 完单

- **POST** `/orders/{orderNo}/arrive`
  - `order.arrive`：`ACCEPTED → ARRIVED`
- **POST** `/orders/{orderNo}/start`
  - `order.start`：`ARRIVED → STARTED`
- **POST** `/orders/{orderNo}/finish`
  - 请求体（MVP）：

```jsonc
{
  "distanceKm": 12.3, // 本次行程实际里程（公里，MVP 可由司机端上报或后端计算）
  "durationMin": 26 // 本次行程实际时长（分钟，MVP 可由司机端上报或后端计算）
}
```

服务内调用顺序（建议）：
1) `calculate.settle`：计算 `finalAmount`（基于 fare_rule + metrics）
2) `order.finish`：`STARTED → FINISHED` + 写入 `final_amount` + 写事件

## 5. 核心服务对内 API（建议在 order-service 提供）

> 这里列的是“被 BFF 调用”的核心接口形态。你已有部分订单查询/事件能力，可在此基础上补齐写接口。

统一前缀（示例）：`/api/v1`

### 5.1 创建订单

- **POST** `/orders`
  - 创建 `trip_order`
  - 写 `order_event(CREATE)`
  - 返回 `orderNo`

### 5.2 指派司机（MVP）

- **POST** `/orders/{orderNo}/assign`
  - 入参：`driverId` / `carId` / `companyId`（以及可选 `etaSeconds`）
  - CAS：只允许 `CREATED → ASSIGNED`
  - 写事件：`ASSIGN`

### 5.3 接单 / 到达 / 开始 / 完单 / 取消

- **POST** `/orders/{orderNo}/accept`
- **POST** `/orders/{orderNo}/arrive`
- **POST** `/orders/{orderNo}/start`
- **POST** `/orders/{orderNo}/finish`
- **POST** `/orders/{orderNo}/cancel`

每个接口建议统一：
- 校验当前状态
- CAS 更新 + 写事件
- 重复调用返回成功（幂等）或返回冲突码

## 6. 轮询与推送：推荐实现方式

### 6.1 MVP 轮询建议（最小闭环）

- 乘客端：`GET /app/api/v1/orders/{orderNo}` 每 2s
- 司机端：`GET /driver/api/v1/orders/assigned?driverId=...` 每 1-2s

### 6.2 升级方案

**方案 1：长轮询**
- `GET /.../watch?sinceVersion=...&timeout=25s`
- 有变化立即返回；否则超时返回空

**方案 2：SSE**
- `GET /.../events/stream`
- 前端 `EventSource` 订阅订单状态变更

**方案 3：WebSocket**
- 单连接推送：新单/状态/位置
- 体验最好，实现复杂度最高

---

## 7. 错误码与返回结构（建议约定）

统一返回：`ResponseVo`。

重点错误：
- **400**：参数校验失败
- **404**：订单不存在
- **409**：状态冲突（例如非 `ASSIGNED` 仍尝试 `accept`；或司机不匹配）

---

## 8. MVP 实施拆分建议（按优先级）

1) `order-service`：补齐 `assign/accept/arrive/start/finish/cancel` 写接口 + CAS
2) `driver-api`：上线/assigned list/accept/arrive/start/finish
3) `passenger-api`：下单（含 assign）/order detail 轮询/cancel
4) `calculate-service`：`estimate/settle`（或先 stub）
5) `map-service`：`route/matrix`（或先 stub）

## 9. 自测记录与「仍为模拟 / 待实现」说明（便于后续迭代）

> 本节用于记录：**当前仓库已能跑通的最小闭环**，以及**仍为占位、假数据或与真实业务有差距**的部分，避免联调时误以为已是生产级能力。

### 9.1 已通过自测（本地联调）

在 **MySQL 各库与 seed 数据就绪**（如 `passenger.customer`、`calculate.fare_rule`、`capacity` 派单演示司机/车辆等）且 **相关服务均已启动** 的前提下，已自测可走通：

- **乘客端**：`POST /app/api/v1/orders` → 返回 `orderNo`；在存在可派单司机时状态可到 **已派单（ASSIGNED）**；`GET /app/api/v1/orders/{orderNo}?passengerId=` 可轮询状态推进。
- **司机端**：`POST /driver/api/v1/drivers/{driverId}/online` → `GET /driver/api/v1/orders/assigned` → `accept` / `arrive` / `start` / `finish` 串联后，订单状态可到 **已完成（FINISHED）**。
- **订单服务**：状态机写操作与事件流水与当前实现一致（详见 `order-service` 代码与库表）。

> **说明**：自测依赖本地/测试环境数据；若派不到单（无满足条件的司机+车），订单会停留在 **CREATED**，属预期，需补 `capacity` 数据或检查司机是否已上线听单。

### 9.2 仍为模拟、简化或与文档「理想形态」不一致的实现（待你后续补齐）

| 领域 | 当前行为 | 后续建议 |
|------|----------|----------|
| **地图 `map-service`** | `POST /api/v1/map/route` 返回 **固定 stub** 里程/时长（如固定米数、秒数），非真实路径规划 | 接入高德/百度等；必要时补 **matrix**（多点到多点的 ETA） |
| **派单 `capacity`「最近司机」** | `nearest-driver` 为 **按城市+车型等条件筛选后的首条候选**，**不是**真实地理距离最近 | 引入司机实时位置、距离排序、并发与拒单策略等 |
| **下单返回里的 ETA** | 乘客侧展示 ETA 往往来自 **路线 stub 的 duration** 等占位，**不是**「司机当前位置 → 上车点」的真实 ETA | 接地图 matrix + 司机坐标 |
| **计费 `calculate`** | 已有 **`estimate`**（预估）；**完单链路未调用 `settle`** | 实现 **`settle`/后结算** 并在 `finish` 前或内写入真实 `final_amount` 规则 |
| **订单 `finish` 金额** | 若未传实付，`order-service` 可用 **`estimated_amount` 兜底** 写入 `final_amount` | 与 `calculate.settle`、优惠/抽成等对齐后再落库 |
| **司机端指派列表 VO** | `GET /driver/api/v1/orders/assigned` 返回中 **`etaSeconds` 等可为 null**（MVP 未算） | 派单或列表接口补充真实 ETA |
| **乘客下单 BFF** | 同步编排（map → estimate → create → dispatch → assign），**非**文档推荐的「对内两段式 + 异步重试指派」 | 按需引入调度/重试、幂等键、超时降级 |
| **分布式追踪** | 各服务可配置 Zipkin 地址；**未起 Zipkin 时** 日志里可能出现上报失败，**一般不影响业务接口** | 本地可关采样或起 Zipkin |
| **长连接 / 推送** | 文档中的 SSE/WebSocket **未实现**；当前靠 **轮询** | 按优先级选型实现 |
| **司机换队等 App 能力** | `capacity` 侧存在 **501 占位**接口（若仍保留） | 按业务文档替换为真实提交/审核流 |

### 9.3 联调时建议固定的最小数据约定（与 seed 脚本对齐）

- 乘客：`passengerId` 与库表 `customer.id` 一致且 `is_deleted=0`。
- 城市/产品线：与 `fare_rule` 的 `province_code` / `city_code` / `product_code`（如 `ECONOMY`）一致，否则 **estimate** 会失败。
- 派单：`driver` 的 `can_accept_order`、`monitor_status`、与 `car` 的 `city_code`、`ride_type_id`、`car_state` 等满足当前 `nearest-driver` 筛选逻辑。
