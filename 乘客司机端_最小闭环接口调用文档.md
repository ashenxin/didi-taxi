# 乘客/司机端最小闭环：接口调用文档（MVP）

本文档描述 **乘客下单 → 系统派单（最近司机）→ 司机接单 → 行程推进 → 完单计费** 的端到端接口调用路线，目标是让 `didi-passenger-h5` / `didi-driver-h5` 可以按本文档完成联调。

**文档角色**：同时包含 **当前仓库已落地的 MVP 行为** 与 **已达成共识的下一期目标设计**（**派单确认窗口**、**司机端 WebSocket + presence**）。目标设计在正文中均标明「规划 / 目标形态」，实现以代码为准；未落地前联调仍以 MVP 为准。

**专项设计（已拍板）**：

- 《`司机端_上线听单与接单设计.md`》—— 状态机细节、调度周期、JWT、WS、订单字段与 `order_event` 等。
- 《`乘客司机端_Redis与听单下线策略.md`》—— **Redis 司机池 / 订单池（规划）**、**司机与上车点距离 ≤3km**、**TTL + 心跳**、**下线/登出删池**、**迟滞匹配（入池为主 + 定时兜底）**、**多单一司机仅接一单**、**下线后 listAssigned 策略 A**；**端到端流程图见该文档 §8**。

适用范围：
- 乘客端 H5：`didi-passenger-h5`（路径示例：`didi-taxi-front/didi-passenger-h5`；经网关调用 `passenger-api`）
- 司机端 H5：`didi-driver-h5`（路径示例：`didi-taxi-front/didi-driver-h5`；经网关调用 `driver-api`）
- 聚合层：`passenger-api(8100)`、`driver-api(8101)`
- 核心服务：`order(8093)`、`capacity(8090)`、`calculate(8091)`
- 地图服务：`map-service(8094)` — **已接入高德 Web 服务 Demo**；H5 **经网关只访问 BFF**，地理编码与驾车规划由 **`passenger-api` 内 Feign 调 map**，浏览器不直连 map（排障除外）。
- **网关**：`gateway(8080)`；校验 `Authorization: Bearer` 后注入 **`X-User-Id`**（值为 JWT `sub`）。客户端**不要**、也**不要尝试**伪造 `X-User-Id`（网关会剥离后再注入）。

## 0. 网关、JWT 与 `X-User-Id`（与当前实现对齐）

### 0.1 浏览器 / H5 该怎么传

| 头 / 行为 | 说明 |
|-----------|------|
| **`Authorization`** | 必填（除登录、发码等**白名单**接口）。格式：`Bearer <accessToken>`。乘客端、司机端各自登录接口返回的 `accessToken`。 |
| **Token 自然过期** | **乘客**：约 **7 天**（登录响应 `expiresIn` 一般为 **604800** 秒，以 `passenger-api` `app.jwt.expiration-seconds` 为准）。**司机**：**8 小时**（**28800** 秒，以 `driver-api` 配置为准）。司机侧 **`token_version` / 登出** 可在 JWT `exp` 到达前使 Token 失效，见《`司机端_登录注册设计.md`》**§4.3**。 |
| **`X-User-Id`** | **不要由前端设置**。由网关在验签通过后写入，下游 BFF 从该头读取当前用户 ID（乘客 `customerId` / 司机 `driverId`）。 |
| **直连 BFF**（`8100` / `8101`） | 无网关时不会自动带 `X-User-Id`，需 Postman 等**手动加** `X-User-Id` 与合法 JWT 场景一致；**联调 H5 请统一走 `http://127.0.0.1:8080`**。 |

### 0.2 网关白名单（无需 Bearer）

- 乘客：`POST /app/api/v1/auth/sms/send`、`login-sms`、`login-password`
- 司机：`POST /driver/api/v1/auth/sms/send`、`register-sms`、`register-password`、`login-sms`、`login-password`（**登出** `POST /driver/api/v1/auth/logout` **须带 Bearer**，不在白名单，见《`司机端_登录注册接口文档.md`》**§9**）
- 管理端：`POST /admin/api/v1/auth/login`（与乘客/司机无关）
- `GET /actuator/health`（健康检查）

其余 `/app/**`、`/driver/**` 在 **`gateway.jwt.require-auth=true`**（默认）下**必须**带合法 JWT；网关按路径选用密钥并校验 **`aud`**（乘客 `app-bff`，司机 `driver-bff`，与各自 BFF 签发配置一致）。

### 0.3 `GATEWAY_JWT_REQUIRE_AUTH=false`（仅本地排障）

关闭强制鉴权时，网关**仍会在请求带合法 Bearer 时**解析 JWT 并注入 `X-User-Id`；若无 Token 或 Token 无效，请求原样转发，BFF 因缺少 `X-User-Id` 返回 **401「未授权，请重新登录」**。因此即使用该开关，**下单等业务仍须带登录拿到的 Token**。

## 1. 关键概念澄清

### 1.1 抢单 vs 派单

- **抢单（broadcast）**：订单发布给一定范围内的多个司机，司机端列表里可见，谁先点击接单谁拿到（需要并发控制与 offer 管理）。
- **派单/指派（dispatch/assign）**：系统选择一个司机（通常是“最近且可接单”），将订单 **指派** 给他；司机端只看到分配给自己的订单。

你当前 MVP 选择：**派单给最近司机**（不做广播抢单）。

### 1.1b 司机池、匹配距离与迟滞匹配（规划，详稿见《`乘客司机端_Redis与听单下线策略.md`》）

与「仅按城市查最近司机」相比，目标形态在派单侧增加：

- **司机池（Redis GEO 等）**：表达当前 **听单中** 司机的 **可匹配位置**；与运力库表 **`monitor_status` 等** 一致，**库表为权威，Redis 为索引**。
- **匹配距离**：司机位置与订单 **上车点** 的 **直线距离 ≤3km**（可配置）方可进入指派候选；超出则 **不匹配**。
- **迟滞匹配**：下单瞬间若无可用司机，订单保持 **`CREATED`**；后续由 **司机入池事件（主）** 与 **定时任务（兜底）** 再尝试「最近 + 距离过滤 + DB `assign`」，**不宜**默认依赖乘客 **高频轮询** 触发匹配（成本高，见《司机池…》**§6**）。

乘客端仍通过 **`GET .../orders/{orderNo}`** 轮询感知状态变化；**状态推进以服务端指派与订单库为准**，轮询仅 **读库**，与《司机池…》**§6.2** 一致。

### 1.1a 派单确认窗口（Offer / 目标产品设计，与 industry 对齐）

与「派单后司机立刻在列表里看到 ASSIGNED」相比，目标产品形态更接近主流网约车：**系统选中候选司机后，先进入「待司机确认」阶段**，司机端弹出 **短时选择窗口（例如约 10s，数值可配置）**：

- **超时是否成立**：**必须由后端裁决**。服务端保存 **`offerExpiresAt`（或 `confirmDeadlineAt`）** 单调递增语义；前端倒计时仅为 UI，**不得**以本地时钟作为是否超时、是否释放订单的唯一依据。
- **司机在时限内确认**：订单进入 **`ACCEPTED`**（或等价状态），派单成立。
- **超时未确认或明确拒绝**：服务端释放该 offer；**先回到 `ASSIGNED`**，**`driver_id` 仍指向本轮超时司机**，改派由 **调度异步扫**（详见《`司机端_上线听单与接单设计.md`》）。确认窗口时长 **10s** 在**配置文件写死**；本阶段**多时区不在范围内**。

**与当前 MVP 代码的关系**：当前实现多为 **同步 `assign` 后直接 `ASSIGNED`**，司机 **`accept` 表示确认**。引入「确认窗口」后，状态机需在 **`ASSIGNED` 与 `ACCEPTED` 之间**插入显式 **`PENDING_DRIVER_CONFIRM`（命名待定）** 或等价语义，并新增/调整 **超时调度任务**（见 §2.3）。**未改表/改代码前，联调仍以现有接口为准。**

### 1.2 轮询 vs 长连接（MVP 与目标形态）

现实中打车软件通常使用长连接推送新订单、状态变化、位置更新；**在线 presence**（是否可交互、心跳是否存活）也依赖长连接或显式心跳，单靠短轮询难以保证「真实在线」。

| 阶段 | 乘客端 | 司机端 |
|------|--------|--------|
| **当前 MVP** | **HTTP 轮询**：`GET /app/api/v1/orders/{orderNo}` 每约 2s，带 **`Authorization`** | **HTTP 轮询**：`GET /driver/api/v1/orders/assigned` 每 1–2s，带 **`Authorization`**；`driverId` Query **可省略**；若传则须与 JWT 一致 |
| **目标形态（规划）** | **WebSocket**（与司机端一致，仅协议层选型；实现分期） | **`WebSocket` + `presence`**：连接即会话通道；心跳与断线由服务端裁决；配合 **派单确认窗口** 推送待确认订单 |

**目标形态（司机端）要点（规划，未在仓库默认实现）**：

- **WebSocket**：用于 **下发「待确认派单」**、订单/状态通知、**应用层心跳**（ping/pong 或业务 heartbeat）；具体 URL、子协议、鉴权方式（如握手带 `Authorization` 或短期 `ticket`）实现时另立补充文档。
- **Presence**：服务端维护「司机是否在线、是否可参与派单/确认」；**以服务端心跳超时与业务规则为准**，客户端倒计时仅作展示。
- **退出登录**：需 **显式下线**（见《司机端_登录注册设计》**退出与 presence**），与 **关闭 WebSocket** 协同，避免「假在线」。

**协议约定**：实时层 **仅 WebSocket**（不并行引入 SSE 作为同类能力）。落地顺序：**MVP 轮询跑通闭环** → **司机侧 WS + presence + 派单确认** → **乘客侧 WebSocket**（分期）。

### 1.3 地图（高德）与调用分层（与当前 `passenger-api` 一致）

**下单闭环（B 层）**已由 **`passenger-api` 内 `MapClient`** 直连 **map-service**：

1. **地理编码（按需）**：若 `origin`/`dest` **未同时提供 `lat`+`lng`**，则用 `address` 或 `name` 拼地址，调用 **`GET /api/v1/map/demo/amap-geocode`** 补全经纬度。BFF 内对 `cityCode` 有**有限映射**（例如 `330100` → 高德 `city=杭州`），其它城市需后续扩展映射表或改传坐标。
2. **驾车规划**：起终点坐标齐备后调用 **`POST /api/v1/map/demo/amap-driving`**（`RouteRequest`），得到 `distanceMeters` / `durationSeconds`，再进入 **`calculate.estimate`** → 建单 → 派单。

| 层级 | 谁调用 | 作用 | 现状 |
|------|--------|------|------|
| **A. H5 选点 / 地图 UI（可选）** | 浏览器（如高德 JS SDK） | 选点、展示、直接拿 lat/lng | 可将坐标传给 BFF，**跳过**服务端 geocode |
| **B. 一步下单（必走 BFF）** | **passenger-api** Feign → map | geocode（缺坐标时）+ 驾车规划 + 预估 | **已接高德 Demo 路径**，非 stub `/map/route` |

H5 **不必**为闭环再调地图 HTTP：只要 BFF 能 geocode（中文地址 + 已支持的城市）或前端传齐经纬度即可。

### 1.4 ETA 是什么（防忘）

- **ETA** = **Estimated Time of Arrival**，中文常称 **预计到达时间**。
- 在本打车场景里，若无特别说明，一般指：**被指派/接单后的司机，从当前位置开到乘客「上车点」预计还要多久**（不是乘客到终点的全程时间，除非接口文档单独写明）。
- 文档或 JSON 里的 **`etaSeconds`**：表示上述 ETA 的 **秒数**（例如 `260` ≈ 4 分 20 秒后到达上车点）。

## 2. 订单状态机（最小）

### 2.1 状态枚举（语义）

- `CREATED`：已创建（下单成功）
- `ASSIGNED`：**已派单**（已写入 **`driver_id`**；目标形态下乘客端统一展示 **「已派单」**，**不**再细分「待司机确认」文案）
- `PENDING_DRIVER_CONFIRM`（目标）：**派单确认窗口内**，等待司机在 **`offerExpiresAt`** 前 **`accept`**
- `ACCEPTED`：司机已接单
- `ARRIVED`：司机已到达上车点
- `STARTED`：行程开始
- `FINISHED`：行程结束（已结算价）
- `CANCELLED`：已取消

### 2.2 允许的流转（MVP）

- `CREATED → ASSIGNED → ACCEPTED → ARRIVED → STARTED → FINISHED`
- `CREATED / ASSIGNED / ACCEPTED` 可 `CANCELLED`（规则后续再细化）
- **待派单超时（已实现）**：`CREATED` 在 **order-service** 配置的 **`order.dispatch.wait-timeout-seconds`（默认 180，即 3 分钟）** 内仍 **未** 进入 `ASSIGNED`（无司机指派），由 **定时任务**（非乘客轮询）系统取消为 **`CANCELLED`**，`cancel_by=系统`，`cancel_reason` 固定为「待派单超时无可用车辆，请稍后重试」；写 **`order_event(ORDER_CANCELLED)`**，`reason_code=DISPATCH_TIMEOUT`。扫描间隔见 **`order.dispatch.timeout-scan-interval-ms`（默认 30000）**。与 **司机池 / 订单池（Redis）无必然清理动作**（`CREATED` 阶段未写入订单池；司机池仅关司机位置）。

### 2.3 允许的流转（目标：派单确认窗口，已拍板）

主路径：

- `CREATED → ASSIGNED → PENDING_DRIVER_CONFIRM → ACCEPTED → ARRIVED → STARTED → FINISHED`

要点：

- **`ASSIGNED → PENDING_DRIVER_CONFIRM`**：进入确认窗口时写入 **`offerExpiresAt`**、**`offer_round`**、**`last_offer_at`**（字段以订单服务为准）；**10s** 窗口由**服务端配置**决定。
- **`PENDING_DRIVER_CONFIRM → ACCEPTED`**：时限内 **`POST .../accept`**（服务端 CAS）；可与 WS 推送并行，**裁决以服务端为准**。
- **`PENDING_DRIVER_CONFIRM` 超时（未 accept）→ `ASSIGNED`**：**`driver_id` 仍指向本轮超时司机**；后续改派/下一轮 offer 由 **每 5s 调度扫描** 推进；**超时、改派**写入 **`order_event`**。
- **乘客取消**：允许 → `CANCELLED`（含待确认阶段）。
- 其余：`ACCEPTED → …`；**不得**仅依赖前端倒计时判定超时。

**乘客侧展示**：统一 **「已派单」**（不细分）；订单详情可返回 **`offerExpiresAt`** 供展示剩余时间（仍以服务端为准）。

**全文细节**：《`司机端_上线听单与接单设计.md`》。

### 2.4 并发与幂等（必须）

所有写操作建议满足：
- **CAS 更新**：`where order_no=? and status=?`（或带 version 字段）
- **幂等**：同一操作重复调用要么返回成功（已是目标状态），要么返回冲突码
- **防串单**：司机侧写操作需校验 `driverId == order.driverId`

## 3. 对外 API（乘客端）—— passenger-api

统一前缀：`/app/api/v1`（经网关的完整 URL 形如 `http://127.0.0.1:8080/app/api/v1/...`）

### 3.0 【特别注明】推荐落地方案（避免分布式事务）

为兼顾后续系统一致性与安全性，本文档采用如下约定（**推荐**）：
- **对外接口形态：一步**（下单一次调用即可）
  - 乘客端只调用一次下单接口，即可拿到 `orderNo`，以及“是否已指派司机”的结果
- **对内实现：两段式**（避免分布式事务）
  - **第一段（强一致，本地事务）**：创建订单为 `CREATED` + 写 `order_event(ORDER_CREATED)`（或 outbox 事件）
  - **第二段（最终一致，可重试）**：找司机并指派，成功则 `ASSIGNED`；失败/超时则保持 `CREATED`，由调度器/重试任务继续处理

**与《乘客司机端_Redis与听单下线策略.md》对齐（规划）**：第二段「补指派」的推荐触发为 **司机入池（主）+ 定时扫描（兜底）**（见该文档 **§6**），**不要**把「乘客每次轮询订单详情」作为默认匹配引擎；若需轮询顺带触发，须 **限频/防抖**。

收益：
- 不引入 2PC/TCC 等分布式事务复杂度
- 更容易做幂等、重试与故障恢复
- 对前端更简单（配合轮询即可看到状态推进）

第一版实现建议：
- 可以先做同步版 `createAndAssign`（同一请求内顺序调用 `capacity` 并 `assign`），但要确保：
  - 幂等（如支持 `Idempotency-Key`）
  - CAS 更新（状态机）
  - 超时降级（找不到司机就返回 `CREATED`，后续由重试/调度器补指派）

当前仓库实现：**同步版** `createAndAssign` 已落地（见 `passenger-api` `PassengerOrderService`），与上表「两段式异步指派」可后续再演进。

专项技术方案（与 **§3.0 对内两段式** 对齐）：《`订单与派单_两段式Outbox与Kafka_技术方案.md`》（对外新增 `POST /app/api/v1/orders/create`；`order-service` Outbox → Kafka；`capacity-service` 消费派单；Top3 候选等细节以该文档为准）。

### 3.0a 乘客端 H5：推荐调用顺序

前提：**经网关 `8080`**，业务请求带 **`Authorization: Bearer <accessToken>`**（见 **§0**）。

1. **发码 / 登录（白名单，可不带 Token）**  
   - `POST /app/api/v1/auth/sms/send` — body：`{ "phone": "13800138000" }`  
   - `POST /app/api/v1/auth/login-sms` — `{ "phone", "code" }`  
   - 或 `POST /app/api/v1/auth/login-password` — `{ "phone", "password" }`  
   响应 `data` 中含 **`accessToken`**、`tokenType`（`Bearer`）、`expiresIn`、`customer`（含 `id` 等）。前端保存 Token，后续请求写入 `Authorization`。
2. **（可选，产品层）选点**  
   H5 用地图 SDK 取 **lat/lng** 可直接写入下单 body；仅中文地址时依赖 BFF **geocode**（见 §1.3 城市映射限制）。
3. **一步下单（需登录）**  
   `POST /app/api/v1/orders/create` — 仅带 **`Authorization`** 即可；**不要**传 `X-User-Id`。
   Body 中 **`passengerId` 可省略**；若填写则**必须与** JWT 对应用户一致，否则 **400**（「请求体中的乘客信息与当前登录身份不一致…」）。服务端最终以 **`X-User-Id`（网关注入）为准** 写入订单。  
   **进行中订单互斥**：若该乘客在库中已有**未删除**且状态**既非「已完单」也非「已取消」**的订单（即除 `FINISHED(5)`、`CANCELLED(6)` 外任意状态），则本次下单返回 **`code=409`**，提示「您已有进行中的订单…」。**H5 应在等待派单 / 行程未结束期间禁用「立即下单」**，并以服务端 **409** 为最终兜底（防多标签页、多设备）。
4. **轮询订单**  
   `GET /app/api/v1/orders/{orderNo}` — 带 **`Authorization`**。若下单响应为 **`CREATED` 且尚未有司机**，前端继续轮询即可；**何时能派上司机由服务端调度/入池匹配完成**（见《`乘客司机端_Redis与听单下线策略.md`》**§6**），**不要**假定仅靠轮询接口会主动「算匹配」。  
   **待派单超时**：若超过 **约 3 分钟**（以 **order-service** `order.dispatch.wait-timeout-seconds` 为准）仍无指派，**order-service 定时任务**会将订单置为 **`CANCELLED`（系统取消）**；轮询读库即可看到状态变化，详情中可带 **`cancelBy` / `cancelReason`**（见 **§3.2**）。H5 可对 **`cancelBy=系统` 且原因含「待派单超时」** 的场景 Toast 提示用户「暂无车辆」类文案。
5. **（可选）取消**  
   `POST /app/api/v1/orders/{orderNo}/cancel` — body 可只含 `cancelReason`；`passengerId` 同样由服务端从 `X-User-Id` 注入，规则同下单。

### 3.1 下单（创建 + 派单 + 返回预估）

**POST** `/app/api/v1/orders`（经网关即路径不变）

**请求头（浏览器）**：`Authorization: Bearer <accessToken>`。`X-User-Id` 由网关注入，**勿在浏览器手动设置**。

**请求体校验要点**：

- `provinceCode`、`cityCode`、`productCode`：必填字符串。
- `origin`、`dest`：必填；子字段 **`name` 必填**；`address` 可选。  
  - 若 **`lat`、`lng` 均缺**：须能通过 `address` 或 `name` 拼出可 geocode 的地址（见 §1.3）。  
  - 若 **`lat`、`lng` 均有**：跳过 geocode，直接驾车规划。

**错误码补充**

- **`409`**：已有进行中订单（`order-service` 在校验通过后创建；`passenger-api` 透传 `msg`）。前端宜展示 `msg` 并停留在当前订单/订单详情页。

**无司机 / 等待派单（与规划文档一致）**

## 4. 对外 API（司机端）—— driver-api

统一前缀：`/driver/api/v1`（经网关完整 URL 形如 `http://127.0.0.1:8080/driver/api/v1/...`；网关转发至 `driver-api:8101`，路径不变）

**鉴权**：业务接口须带 **`Authorization: Bearer`**；网关注入 **`X-User-Id`**（字符串形式的 `driverId`）。路径参数、Query、JSON 里的 **`driverId` 若出现，必须与 `X-User-Id` 一致**，否则 **403「禁止操作其他司机数据」**。

### 4.0 司机端认证（白名单，无需 Token）

| 接口 | 说明 |
|------|------|
| `POST /driver/api/v1/auth/sms/send` | 发码；body：`{ "phone" }` |
| `POST /driver/api/v1/auth/register-sms` | 短信注册并登录；`{ "phone", "code" }` |
| `POST /driver/api/v1/auth/register-password` | 短信验证后设密并登录；`{ "phone", "code", "password" }` |
| `POST /driver/api/v1/auth/login-sms` | 验证码登录 |
| `POST /driver/api/v1/auth/login-password` | 密码登录 |

响应 `data` 中含 **`accessToken`** 等（与乘客端结构类似，字段以 `DriverLoginResponse` 为准）。

**推荐调用顺序（MVP 闭环）**：任选注册/登录拿 Token → `POST .../drivers/{driverId}/online` → 轮询 `GET .../orders/assigned` → `accept` → `arrive` → `start` → `finish`。

**推荐调用顺序（目标形态，规划）**：登录拿 Token → 建立 **WebSocket** 并完成 **presence 上线**（与 `online` 业务含义对齐，细节见 §4.5）→ 经 WS 或 HTTP 接收 **待确认派单** → 在 **`offerExpiresAt` 前**调用确认接口（超时仅服务端生效）→ 后续 `arrive` / `start` / `finish` 仍以 REST 或统一信令为准。

### 4.1 司机上线/下线（影响派单候选集）

**POST** `/driver/api/v1/drivers/{driverId}/online`

**请求头**：`Authorization: Bearer <token>`；`{driverId}` 须与 JWT / `X-User-Id` 一致。

请求体：

```jsonc
{
  "online": true // 是否上线（true 上线可派单，false 下线不参与派单）
}
```

服务内调用：`capacity` 更新 `monitor_status`（0/1/2 口径按你现有字段）。

### 4.2 拉取“分配给我的订单”（派单模式）

**GET** `/driver/api/v1/orders/assigned`  
可选 Query：**`driverId`** — 省略时默认使用 **`X-User-Id`** 对应司机；传入时须与登录身份一致。

**请求头**：`Authorization: Bearer <token>`。

响应（示例）：

```jsonc
{
  "code": 200, // 业务码
  "msg": "success", // 提示信息
  "data": [
    {
      "orderNo": "20260330xxxx", // 订单号
      "status": "ASSIGNED", // 订单状态（通常是已指派 ASSIGNED）
      "pickup": {
        "name": "西湖景区" // 上车点名称（可扩展经纬度/地址）
      },
      "etaSeconds": 260 // 预计到达上车点时间（秒）
    }
  ]
}
```

服务内调用：`order.listAssignedToDriver`（按 `driver_id` + `status in (ASSIGNED, PENDING_DRIVER_CONFIRM)` 查询）。**`ASSIGNED` 与 `PENDING_DRIVER_CONFIRM` 均可出现在同一列表**，司机刷新即可拉多笔待确认单。

**与「服务中」互斥（order-service）**：司机已有 **`ACCEPTED`～`STARTED`** 订单时，**不再**对该司机执行 **`assign`**（乘客下单链路会得到 **409**）；**`FINISHED` / `CANCELLED`** 后恢复可派。司机 **`accept` 一单** 后，同司机其余 **`ASSIGNED` / `PENDING_DRIVER_CONFIRM`** 单由系统 **自动取消**（原因「司机已接其他订单」）。详见《`司机端_上线听单与接单设计.md`》**§2.5**。

**司机端 H5（`didi-driver-h5`，路径示例：`didi-taxi-front/didi-driver-h5`）**

- **指派列表**：「刷新指派单」调用 **`GET .../orders/assigned`**；列表可同时含 **`ASSIGNED` / `PENDING_DRIVER_CONFIRM`**。当 **≥2 条** 时展示提示：**确认其中一单后，其余待确认单由系统自动取消**。
- **行程跟单中、列表为空**：若已 **`accept`** 并进入行程面板轮询，且当前无新指派，展示 **「暂无新指派单」** 说明（**已接单～行程中** 不会再对该司机 **`assign`**，与上节一致）。
- **409**：接口若返回 **`code=409`**（并发冲突等），宜 **Toast** 展示服务端 **`msg`**（与 **`src/api/http.js`** 抛出的 `ApiError` 对齐）；**`accept` 成功**后可 Toast「已接单」。
- 实现参考：**`src/App.vue`**（`loadAssigned`、`acceptOrder`、`isConflictError`）、**`src/utils/orderStatus.js`**（列表状态文案）。

### 4.3 接单（确认指派）

**POST** `/driver/api/v1/orders/{orderNo}/accept`

**请求头**：`Authorization: Bearer <token>`。

请求体（`driverId` 须等于当前登录司机）：

```jsonc
{
  "driverId": 80001
}
```

服务内调用顺序（建议）：
1) `capacity.checkDriverCanAccept`：`can_accept_order=1` 且在线
2) `order.accept`：CAS（MVP：`ASSIGNED → ACCEPTED`；目标：增加 **`PENDING_DRIVER_CONFIRM → ACCEPTED`**，且 `driver_id` 必须匹配；见《`司机端_上线听单与接单设计.md`》）

### 4.4 到达 / 开始 / 完单

均需 **`Authorization`**；body 中 **`driverId` 与登录身份一致**（`finish` 成功后会以服务端为准写回 `driverId`）。

- **POST** `/driver/api/v1/orders/{orderNo}/arrive` — body：`{ "driverId" }`；`ACCEPTED → ARRIVED`
- **POST** `/driver/api/v1/orders/{orderNo}/start` — body：`{ "driverId" }`；`ARRIVED → STARTED`
- **POST** `/driver/api/v1/orders/{orderNo}/finish` — body（MVP）：

```jsonc
{
  "driverId": 80001,
  "distanceKm": 12.3,
  "durationMin": 26,
  "finalAmount": null
}
```

`finalAmount` 可选；`distanceKm` / `durationMin` 可选，按订单服务与计费演进使用。

服务内调用顺序（建议）：
1) `calculate.settle`：计算 `finalAmount`（基于 fare_rule + metrics）
2) `order.finish`：`STARTED → FINISHED` + 写入 `final_amount` + 写事件

### 4.5 规划：司机端 WebSocket、presence 与派单确认窗口

> **本节为摘要**；**已拍板参数与字段**见《`司机端_上线听单与接单设计.md`》。

**目标**：

1. **WebSocket**：**仅**采用 WS 作为实时协议；握手 **`Authorization: Bearer`**；**审核未通过不允许建连**（HTTP 仍可登录查看原因）。
2. **Presence**：心跳 **15s**、**连续 3 次**未应答 → 离线并**写库**；与 **10s 派单确认**独立。
3. **会话**：同一手机号 **全局唯一有效 JWT**（**`token_version` + Redis**）；**登出强制作废**；**单终端**（新踢旧）。
4. **派单确认**：复用 **`POST .../accept`**；消息带 **`messageId`**，服务端去重缓存 **8h**；HTTP 与 WS **字段名一致**。
5. **部署**：优先 **`driver-api` 内嵌 WS**（最小工作量）；网关需支持 WS 与**多实例**演进。

**退出登录**：关闭 WS + 调用 **`/auth/logout`**；待确认 offer **不再发给该司机**（见上线听单文档）。

**观测**：本期不做专项监控大盘。

统一前缀（示例）：`/api/v1`

  - 写 `order_event(CREATE)`

- **POST** `/orders/{orderNo}/arrive`
- **POST** `/orders/{orderNo}/start`
- **POST** `/orders/{orderNo}/finish`
- **POST** `/orders/{orderNo}/cancel`

每个接口建议统一：
- 校验当前状态
- CAS 更新 + 写事件
- 重复调用返回成功（幂等）或返回冲突码

## 6. 轮询、推送与实时层（MVP 与目标）

### 6.1 MVP：HTTP 轮询（当前）

- **乘客端**：`GET /app/api/v1/orders/{orderNo}` 每约 2s，带 **`Authorization`**
- **司机端**：`GET /driver/api/v1/orders/assigned` 每 1–2s，带 **`Authorization`**（`driverId` Query 可省略）

### 6.2 目标：司机端 WebSocket + presence（规划）

- **WebSocket**：推送 **待确认派单**、订单状态；承载 **应用层心跳**（服务端判活、维护 presence）。
- **presence**：与「上线听单」「可交互」绑定；**断线/心跳超时/登出** 更新离线，详见 **§4.5** 与《`司机端_上线听单与接单设计.md`》。
- **乘客端**：目标形态为 **WebSocket**（与司机端协议层一致）；过渡期内可继续 HTTP 轮询；订单 VO 可返回 **`offerExpiresAt`**（仍以服务端为准）。

### 6.3 其他可选形态（非主路径）

- **长轮询**：仅作备选；**实时层主选 WebSocket**。

## 7. 错误码与返回结构（建议约定）

统一返回：`ResponseVo`（`code` / `msg` / `data`）；成功 **`code === 200`**。

重点错误：
- **400**：参数校验失败；乘客端 body **`passengerId` 与登录身份不一致**
- **401**：未带 Token、Token 无效/过期；或 BFF **未收到 `X-User-Id`**（常见：未走网关、或关鉴权但未带 Bearer）
- **403**：乘客看他人订单；司机 **`driverId` 与 JWT 不一致**
- **404**：订单不存在
- **409**：状态冲突（例如非 `ASSIGNED` 仍尝试 `accept`）

**规划：派单确认窗口**：在 **`offerExpiresAt` 之后**再调用确认 → 应返回 **409 或业务码**（如「确认已超时」），以**服务端状态**为准，而非前端倒计时。

## 9. 自测记录与「仍为模拟 / 待实现」说明（便于后续迭代）

> 本节用于记录：**当前仓库已能跑通的最小闭环**，以及**仍为占位、假数据或与真实业务有差距**的部分，避免联调时误以为已是生产级能力。

### 9.1 已通过自测（本地联调）

在 **MySQL 各库与 seed 数据就绪**（如 `passenger.customer`、`calculate.fare_rule`、`capacity` 派单演示司机/车辆等）且 **网关 + 相关服务均已启动** 的前提下，已自测可走通：

- **乘客端**：登录拿 Token → 经 **`8080`** `POST /app/api/v1/orders/create`（`Authorization`，body 可不传 `passengerId`）→ 返回 `orderNo`，状态通常为 **`CREATED`**；派单异步推进后可到 **`PENDING_DRIVER_CONFIRM`**；`GET /app/api/v1/orders/{orderNo}` 轮询状态推进。
- **司机端**：登录 → `POST /driver/api/v1/drivers/{driverId}/online` → `GET /driver/api/v1/orders/assigned` → `accept` / `arrive` / `start` / `finish` 串联后，订单状态可到 **已完成（FINISHED）**。
- **订单服务**：状态机写操作与事件流水与当前实现一致（详见 `order-service` 代码与库表）。

> **说明**：自测依赖本地/测试环境数据；若派不到单（无满足条件的司机+车），订单会停留在 **CREATED**，属预期，需补 `capacity` 数据或检查司机是否已上线听单。

### 9.2 仍为模拟、简化或与文档「理想形态」不一致的实现（待你后续补齐）

| 领域 | 当前行为 | 后续建议 |
|------|----------|----------|
| **地图 `map-service` + BFF** | `passenger-api` 已 Feign：**geocode**（`GET .../amap-geocode`）+ **驾车**（`POST .../amap-driving`） | 扩展 **`cityCode → geocode city`** 映射；必要时 **matrix** 做真实司机 ETA |
| **派单 `capacity`「最近司机」** | `nearest-driver` 为 **按城市+车型等条件筛选后的首条候选**，**不是**真实地理距离最近 | 引入司机实时位置、距离排序、并发与拒单策略等 |

| **长连接 / 推送** | 当前 **未实现**；司机端靠 **HTTP 轮询** | **目标**：**仅 WebSocket** + presence（§4.5、§6.2、《`司机端_上线听单与接单设计.md`》）；乘客端同期目标为 WS |
| **派单确认窗口** | 当前多为 **`assign` 后直接 `ASSIGNED`**，司机 `accept` | **目标**：**`CREATED→ASSIGNED→PENDING_DRIVER_CONFIRM`**，超时回 **`ASSIGNED` 且保留 `driver_id`**，**5s** 调度扫，见专项文档 |
| **司机换队等 App 能力** | `capacity` 侧存在 **501 占位**接口（若仍保留） | 按业务文档替换为真实提交/审核流 |

### 9.3 联调时建议固定的最小数据约定（与 seed 脚本对齐）

- 乘客：登录用户 JWT `sub` / 网关注入的 `X-User-Id` 与库表 **`customer.id`** 一致且 `is_deleted=0`（body 可不传 `passengerId`）。
- 城市/产品线：与 `fare_rule` 的 `province_code` / `city_code` / `product_code`（如 `ECONOMY`）一致，否则 **estimate** 会失败。
- 派单：`driver` 的 `can_accept_order`、`monitor_status`、与 `car` 的 `city_code`、`ride_type_id`、`car_state` 等满足当前 `nearest-driver` 筛选逻辑。
