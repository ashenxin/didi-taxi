# 司机端 WebSocket 与实时协议：名词快速入门

本文用**日常语言**解释：做「司机长连接 + 派单确认」时经常听到的 **WS URL、握手鉴权、消息 schema、订单状态字段** 等到底是什么、和 HTTP 有什么不同。  
**不是最终实现规范**（具体路径、JSON 字段以落地 PR 为准），目的是让你和研发对齐时**少踩概念坑**。

---

## 1. 先建立一张「地图」：HTTP vs WebSocket

| | **HTTP（你们现在用的接口）** | **WebSocket（计划中的长连接）** |
|---|------------------------------|--------------------------------|
| **像什么** | 浏览器每次「打电话问一句、挂掉」：一问一答 | 拨通后**一直通着**，双方随时说话 |
| **典型用途** | 登录、拉订单列表、点「确认接单」按钮提交一次 | **推送**「来新单了」、**心跳**「我还活着」、减少疯狂轮询 |
| **地址** | `https://.../driver/api/v1/...` | `wss://...` 开头（加密），后面跟**路径**，见下文 **WS URL** |

一句话：**HTTP 适合「办事」；WebSocket 适合「保持联系 + 实时通知」。** 业务上两者会**同时存在**：登录仍用 HTTP，长连接用来收单和保活。

---

## 2. WS URL 是什么？

**URL** = 资源地址。平时见的是 `http://` 或 `https://`。

**WebSocket 的 URL** 用专用协议名：

- **`ws://`**：不加密（一般只用于本地开发）
- **`wss://`**：加密（生产环境应使用，和 HTTPS 类似）

**例子（虚构，仅帮助记忆形状）：**

```text
wss://api.example.com/driver/ws/v1/connect
```

- **`wss://`**：协议
- **`api.example.com`**：域名（可能和网关、CDN 同一域名，也可能单独子域）
- **`/driver/ws/v1/connect`**：**路径**，表示「这是司机端用的、第 1 版 WebSocket 接入点」

**你需要理解的点：**

- WS URL **不是**随便字符串，和 HTTP 一样要由**后端/网关配置**好，前端按文档连接。
- 很多项目会把 WS **挂在网关后面**（例如 `wss://127.0.0.1:8080/driver/ws/...`），和 REST 共用 8080，但**路径不同**。

## 3. 「握手」和「握手鉴权」是什么？

### 3.1 握手（Handshake）

浏览器（或 App）想建 WebSocket 时，**先发一个特殊的 HTTP 请求**，问服务器：「能不能升级成 WebSocket？」  
服务器若同意，返回 **101 Switching Protocols**，之后**同一条 TCP 连接**就变成 WebSocket，后面传的就是「帧」而不是普通 HTTP 页面。

这段升级过程就叫 **WebSocket 握手**。

### 3.2 握手鉴权（在握手时证明「我是已登录司机」）

REST 接口里，你们习惯在**每个请求头**里带：

```http
Authorization: Bearer <JWT>
```

WebSocket **也可以**在**第一次握手**的请求里带同样的头（是否支持取决于网关/服务器配置）。这样服务器在**同意连上之前**就知道你是谁，**拒绝未登录或 token 无效**的连接。

**常见几种做法（概念层面）：**

| 方式 | 含义 | 类比 |
|------|------|------|
| **握手带 `Authorization: Bearer`** | 升级请求里就带 JWT | 进门先亮工牌 |
| **Query 带短期票据** | `wss://.../connect?ticket=xxx`，ticket 是服务端刚发的短期令牌 | 临时通行证 |
| **先 HTTP 换 ticket 再连 WS** | `POST` 拿 `wsTicket`，再连 WS | 先去前台换访客贴纸 |

**你需要理解的点：**

- **握手鉴权** = **在连接建立时**完成身份校验，避免谁都能连上你们的 WS。
- 具体用 Bearer 还是 ticket，是**实现选型**，但目的都是：**长连接也要和登录态绑定**。

## 4. 消息 schema 是什么？

**Schema** 原意是「结构、纲要」。在接口语境里：

- **消息 schema** = **约定好「发什么 JSON、字段叫什么、类型是什么」** 的一张「合同」。

没有 schema，就会出现：A 写 `type`，B 写 `msgType`，前端解析全乱。

**极简例子（纯属教学，非最终定稿）：**

**服务端 → 司机：新派单待确认**

```json
{
  "v": 1,
  "type": "OFFER_NEW",
  "payload": {
    "orderNo": "202604110001",
    "offerExpiresAt": "2026-04-11T12:00:10+08:00",
    "pickupName": "杭州东站"
  }
}
```

**司机 → 服务端：心跳**

```json
{
  "v": 1,
  "type": "PING",
  "payload": {
    "clientTime": "2026-04-11T12:00:00+08:00"
  }
}
```

约定好：

- 最外层必有 **`type`**（消息类型）
- 业务数据放 **`payload`**
- **`v`** 表示协议版本，以后大改可以加 `v: 2`

这就是一种 **schema 思路**（名字可以叫 envelope、frame，不重要，重要的是**统一结构**）。

**`messageId`（已拍板）**：需要幂等或防重复处理的消息应带 **`messageId`**；服务端用 **Redis** 记录已处理 ID，**保留 8 小时**（TTL），避免重连、重试导致重复执行业务；也为后续 **防疲劳策略** 留扩展空间。详见《`司机端_上线听单与接单设计.md`》**§5.5** 与《`司机端_登录注册设计.md`》**§8.1**。

**你需要理解的点：**

- **消息 schema** = **长连接里每条消息长什么样** 的约定。
- 和 HTTP 的「请求体 / 响应体」一样，只是**方向更自由**（服务端可主动推）。

## 5. 「订单状态字段名」在吵什么？

你们文档里已经有业务状态：**CREATED、ASSIGNED、ACCEPTED** 等。

**字段名**问题出在：**数据库 / 订单服务 / 给前端的 JSON** 里，这个状态**属性到底叫什么**、**值长什么样**。

常见几种风格：

| 风格 | 例子 | 特点 |
|------|------|------|
| **蛇形** | `order_status` | 数据库列名常见 |
| **驼峰** | `orderStatus` | Java/JavaScript JSON 常见 |
| **枚举对象** | `status: { "code": 1, "en": "ASSIGNED", "zh": "已派单" }` | 你们 passenger-api 已有类似形态 |

**「派单确认」进来以后**，往往还会多一个阶段，例如文档里的 **`PENDING_DRIVER_CONFIRM`**。  
这时要统一说清楚：

- **数据库**里字段名（如 `status`）
- **API 返回**里叫 `status` 还是 `orderStatus`
- **数值**用 0/1/2 还是字符串 `"ASSIGNED"`

**你需要理解的点：**

- **订单状态字段名** = 在 JSON 里**哪个 key** 表示状态，以及**允许哪些取值**。
- 这和 WebSocket **无必然关系**：HTTP 订单详情和 WS 推送里，**最好沿用同一套**，避免司机端两套解析逻辑。

## 6. 和「服务端裁决超时」怎么串起来？

你们已定：**倒计时是否过期，以服务端为准**。

在协议层通常体现为：

- 每条 **待确认派单** 消息里带 **`offerExpiresAt`**（**绝对时间**，建议 ISO 8601 带时区）
- 司机点「接单」时，服务端查库：**当前时间是否仍早于 `offerExpiresAt` 且订单仍处于待确认状态**  
  → 通过则 `ACCEPTED`；否则返回「已超时」类错误

前端本地可以显示 **10、9、8…**，那只是 **UI**，**不能**当法律依据。

## 7. 一张极简时序（帮助脑补）

```text
司机 App                    网关 / 服务
   |                              |
   |--- WebSocket 握手 + JWT ---->|
   |<-------- 101 升级成功 --------|
   |                              |
   |<--- OFFER_NEW (offerExpiresAt)|
   |                              |
   |--- HTTP 或 WS: 确认接单 ----->|  （服务端校验截止时间）
   |<-------- 200 / 409 -----------|
```

（确认接单用 HTTP 还是 WS 发第二条消息，是**实现二选一**，文档级可先不锁死。）

## 8. 读完你可以怎么和研发对话？

你可以直接问三句：

1. **WS URL** 定成 `wss://谁/哪条路径`？本地怎么连？
2. **握手**用 Bearer 还是 ticket？token 过期了连接怎么处理？
3. **消息 schema**：先发一页「type 列表 + payload 示例」，尤其是 **OFFER_NEW** 和 **心跳**。

订单侧再问一句：

4. **订单状态**在订单详情和 WS 推送里是不是**同一套字段和枚举**？**`PENDING_DRIVER_CONFIRM`** 在 JSON 里最终叫什么？
