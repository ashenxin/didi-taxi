# 司机端：上线、听单与接单（目标设计）

本文档固化 **司机端实时通道 + 派单确认 + 会话与调度** 的**已拍板结论**，与《`乘客司机端_最小闭环接口调用文档.md`》《`司机端_登录注册设计.md`》《`司机端_WebSocket与实时协议入门.md`》交叉引用。  
**当前仓库代码可能尚未实现**；实现以 PR 为准，本文用于需求对齐与评审。

---

## 1. 文档范围

- **上线 / 听单**：`monitor_status`、presence、WebSocket、心跳。
- **接单**：派单确认窗口、`POST .../accept` 复用、状态机、调度扫描。
- **会话**：同一手机号全局唯一有效 JWT、`token_version` + Redis、登出强制作废。
- **不包含**：具体类名、网关 XML 片段（见网关技术文档与实现 PR）。

## 2. 订单状态机（目标）

### 2.1 主路径

```text
CREATED → ASSIGNED → PENDING_DRIVER_CONFIRM → ACCEPTED → ARRIVED → STARTED → FINISHED
```

- **`ASSIGNED`**：已写入 **`driver_id`**，乘客端展示 **「已派单」**（**不**再细分「待司机确认」文案）。
- **`PENDING_DRIVER_CONFIRM`**：已进入 **派单确认窗口**，服务端写入 **`offerExpiresAt`**、**`offer_round`**、**`last_offer_at`** 等；司机需在窗口内 **`accept`**。
- **确认窗口时长**：**10s**，在**配置文件写死**（本阶段不按城市/产品线区分；**不考虑多时区**，时间以服务端为准）。

### 2.2 超时（未在 `offerExpiresAt` 前 `accept`）

- **先回到 `ASSIGNED`**（**不**直接清空业务语义上的「已派单」）。
- 订单上的 **`driver_id` 仍指向本轮超时那位司机**；后续 **改派 / 下一轮 offer** 由 **调度异步任务** 推进（更新 `driver_id`、`offer_round` 等后再进入 `PENDING_DRIVER_CONFIRM`）。
- **超时、改派**均写入 **`order_event`** 流水，便于审计与排障。

### 2.3 乘客取消

- 在 **`PENDING_DRIVER_CONFIRM`（及前置状态）** 允许乘客取消 → **`CANCELLED`**（规则与现有取消域一致）。

### 2.4 司机侧 `accept`

- **复用** **`POST /driver/api/v1/orders/{orderNo}/accept`**（或 order-service 对内等价接口）；合法状态需包含 **`PENDING_DRIVER_CONFIRM`**（及迁移期兼容策略，由实现定）。

---

## 3. 调度与幂等

- **异步扫描周期**：**每 5s** 扫描待处理订单（`ASSIGNED` / `PENDING_DRIVER_CONFIRM` 等，具体 WHERE 条件由实现定）。
- **幂等**：同一订单、同一事件不得重复推进；使用 **CAS + `order_event`** 或分布式锁，避免多实例重复执行。
- **司机离线**：**capacity / presence** 尽快落库；**订单状态推进**以 **调度扫描** 为主（与「断线即更新司机维度」配合）。

---

## 4. 会话与 JWT（同一手机号全局一个有效 Token）

- **策略**：**`token_version` + Redis**（或等价：登录成功递增版本，JWT claims 携带 **`token_version`**，网关或 BFF 校验与 Redis 当前值一致）。
- **语义**：同一手机号 **新登录** 使 **旧 JWT 全局失效**；旧终端 HTTP **401**、WebSocket **断开或握手失败**。
- **登出**：**`POST /driver/api/v1/auth/logout`** **强制作废会话**（递增 `token_version` 或列入失效集合）；**待确认 offer 不再发给该司机**（订单侧按规则剔除或等待调度处理）。
- **Redis**：键命名、TTL 与 **`司机端_登录注册设计.md` §8** 一并规划（在 `driver:*` 前缀下增加 `driver:token:ver:{driverId}` 等，实现时细化）。

## 5. WebSocket 与 presence

### 5.1 协议选型

- **实时层仅使用 WebSocket**（**不**并行引入 SSE 作为同类能力）。
- **乘客端**后续也改为 **WebSocket**（另文/另迭代；与司机可共用网关路径规范或分路径）。

### 5.2 部署形态（最小工作量）

- **优先**：在 **`driver-api` 进程内** 提供 WebSocket Endpoint（**最小拆分**、单仓库联调快）。
- **网关**：**必须**支持 **WebSocket 升级** 与转发；**后续多实例**时，通过 **Sticky** 或 **Redis Pub/Sub（或消息队列）** 解决跨节点推送，实现阶段选型。

### 5.3 握手鉴权

- **握手携带 `Authorization: Bearer <JWT>`**（与 HTTP 一致）；**未通过审核不允许建立 WS**（握手直接拒绝），业务规则最简单；HTTP 仍可用于登录后查看原因/补件。

### 5.4 心跳与离线

- **间隔 15s**；**连续 3 次**未收到应用层心跳/应答 → **判离线**，**写数据库**（如 `monitor_status` / 在线标记），**不能接单**。
- **与派单 10s 倒计时独立**，互不替代。
- **单终端**：**新连接踢旧连接**；与 **全局唯一 JWT** 一致。

### 5.5 消息 envelope

- **统一格式**（`v` / `type` / `payload` 等），**HTTP 订单字段与 WS 推送字段同一套命名**。
- **`messageId`**：每条可重放消息携带；服务端 **Redis 去重**，**保留 8 小时**（为后续 **防疲劳 / 重复点击** 等留扩展空间）。

## 6. 订单表与流水（建议字段）

| 字段 / 流水 | 说明 |
|-------------|------|
| **`offer_expires_at`** | 当前确认窗口截止时间（服务端写入）。 |
| **`offer_round`** | 第几轮 offer（同司机重复派单时递增）。 |
| **`last_offer_at`** | 最近一次发起 offer 的时间。 |
| **`order_event`** | **超时、改派、确认、取消** 等均记流水，类型与 payload 实现时枚举。 |
