# 后台管理系统：权限清单与鉴权设计

本文档整理 **后台管理端（`didi-Vue` + `admin-api` + `passenger`）** 在权限、登录、数据范围、审计等方面的**已定方案与实施要点**，便于开发与联调对齐。  
**网关、分布式配置中心**等见文末「后续演进」，当前阶段不展开实现。

---

## 1. 文档范围

| 包含 | 不包含（当前阶段） |
|------|---------------------|
| 角色与菜单（权限清单）、JWT、Redis 会话摘要 | 网关统一鉴权（上线网关后再定） |
| 数据范围（省/市）在 `admin-api` 的统一约束 | 乘客/司机端登录 |

| 操作日志（切面） | OAuth2 / 第三方登录 |

## 2. 架构边界

- **`admin-api`**：对外暴露后台 BFF；**签发并校验 JWT**；**不直连 `passenger` 库**；通过 **Feign 调用 `passenger` 服务** 完成 `sys_*` 用户/角色/菜单的读写与校验。
- **`passenger` 服务**：**唯一**访问 `passenger` 库中 **`sys_*` 表**；负责持久化与 **省/市与角色一致性** 等业务校验（写入时代码校验）。
- **首期鉴权**：**仅由 `admin-api` 校验 JWT**（无独立网关验签）。

---

## 3. 角色与数据范围（权限清单的业务语义）

| 角色 | 菜单（权限清单） | 数据范围 |
|------|------------------|----------|
| **超级管理员** | **代码侧**（`AdminSysMenuService`）取 **全部有效菜单**；`sys_role_menu` 可冗余挂全量便于审计/报表，**解析菜单树时不依赖** `sys_role_menu` | 不按省/市过滤（`province_code`、`city_code` 均为空） |
| **省份管理员** | 与超管相同：**全部有效菜单**（与 `sys_role_menu` 是否逐条挂接无关，**解析时同样取全量**）；数据范围由接口按省裁剪 | **单一省份**：`province_code` 必填，`city_code` 为空；多省需求通过 **换账号** 解决 |
| **城市操作员** | **代码侧**在全部有效菜单上 **排除 `path` 以 `/system` 开头的节点**（含系统管理及其子菜单） | **单一城市**：`province_code`、`city_code` 均有值 |

**说明（与实现对齐）**：内置三角色 **`SUPER` / `PROVINCE_ADMIN` / `CITY_OPERATOR`** 的菜单树由 **`AdminSysMenuService` 按角色编码分支解析**，**不读取** `sys_role_menu` 并集；表中 `sys_role_menu` 仍建议与种子脚本保持一致，便于审计或后续扩展「自定义角色」走 `sys_role_menu`。

**菜单级授权**：当前以 **能进入哪些菜单** 为主；`sys_menu.perms`（含 `visible=0` 的按钮行）供前端按钮显隐；**首期接口鉴权以 JWT + 数据域为主**，`perms` 与 `didi-Vue` 约定一致即可。

**管理员账号维护**：**超级管理员** 可创建 **省管理员** 或 **市操作员**；**省管理员** 仅可在 **本人所属省** 下创建/维护 **市操作员**（`CITY_OPERATOR`），落库省份以服务端 **`caller` 的 `province_code` 为准**，详见《权限与接口文档》**§4.5**。

## 4. 数据模型摘要（`passenger` 库，`sys_` 前缀）

- **`sys_user`**：登录名、密码摘要、`province_code` / `city_code`、`status`、软删；**软删后用户名可复用**（`username_active` 生成列 + 唯一索引，见已定 DDL）。
- **`sys_role`**：角色编码（如 `SUPER` / `PROVINCE_ADMIN` / `CITY_OPERATOR`）等。
- **`sys_user_role`**：用户与角色 **多对多**（首期可一人一角色一行）。
- **`sys_menu`**：树形菜单；含 **`path`、`name`、`icon`、`component`、`perms`、`sort`、`visible`、`status`** 等；**无独立 `sys_permission` 表**，按钮级权限串写在 **`perms`**（常配合 **`visible=0`** 子行）。
- **`sys_role_menu`**：角色与菜单关联，构成 **权限清单（可见菜单集合）**。

**首个超级管理员**：**手动录入**（SQL 或运维工具），不设自动化 bootstrap 接口（已定）。

## 5. JWT 与 Redis

### 5.1 JWT（`admin-api` 签发）

- 传输：**仅 HTTP Header** `Authorization: Bearer <token>`。
- **必须包含 `exp`**（过期时间）；`sub` 建议为 **`sys_user.id`**。
- **踢下线字段**：在 JWT 中增加 **`token_version`（或 `perm_version`）** 整型；与 **`sys_user` 中持久化字段**（如 `token_version`）一致时 token 才有效。  
  **改权限 / 改角色 / 改省市区 / 禁用用户 / 改密** 等敏感变更后：**递增该用户 `token_version`**，使已签发 JWT 立即失效。
- **密钥**：**`jwt.secret` 从环境变量读取**；二期再迁分布式配置中心。

### 5.2 Redis

- **用途**：缓存 **`userId → 角色、省/市、菜单 id 列表等****鉴权摘要**，减少每次请求打 `passenger`。
- **Key 示例**：`adm:user:{userId}`（具体结构与团队统一）。
- **TTL**：建议设置 **过期时间**（短于或配合 JWT 寿命）；**权限或 `token_version` 变更时删除该 key**。
- **注意**：Redis 仅存摘要；**不以 Redis TTL 替代 JWT `exp`**；**密码、完整 token 不落 Redis**。

**实现（仓库）**：`admin-api` 使用 `AdminSecurityContextRedisCache` + `StringRedisTemplate`，配置项 `admin.security-context-cache.*`（默认前缀 `adm:user:`、TTL 1800s）；`JwtAuthenticationFilter` 先读缓存再 Feign。`passenger` 在 `AdminSysStaffService` 更新/软删用户后通过 `AdminSecurityContextCacheEvictor` 删同键（`passenger.admin.security-context-cache.key-prefix` 须与前者一致）。本地需运行 Redis；`admin.security-context-cache.enabled=false` 或未注入 `StringRedisTemplate` 时退回仅 Feign。

## 6. 数据权限（列表/导出/统计）

- **菜单只控制「能否进页面」**；**列表类接口必须按省/市过滤**，避免越权看数。
- **在 `admin-api` 统一封装**「当前登录用户的数据范围」（从 JWT + Redis 或回源 `passenger` 解析）：超级管理员不加省/市条件；省管员加 `province_code`；市操作员加 `province_code` + `city_code`。
- **所有后台数据查询**（订单、运力、报表等）须 **经同一套约束**（注解 + AOP 或统一 Query 参数注入），避免单个 Controller 遗漏。

### 6.1 实现现状

**`admin-api`**：`AdminDataScope`；已接入订单、计价、运力、换队（详见《权限与接口文档》**§4.7**）。**403** = 查询参数越域；**404** = 资源不在域内（与「不存在」不分）。新列表/导出面相同约束；长期仍可落到 §6 所述 AOP。

## 7. 操作日志（审计）

- **实现方式**：在 `admin-api` **增加一层切面（AOP）**，对约定注解或 Controller 包路径拦截，记录：
  - 操作人（`userId` / `username`）、时间、请求路径与方法、核心参数摘要、结果成功/失败等。
- **原则**：**不记录密码**、不记录完整 JWT；敏感字段脱敏。
- **存储**：表结构或 ES 等由实现阶段定；本文档仅要求 **具备可追溯审计能力**。

---

## 8. 登录与安全（首期范围）

- **首期不实施**：登录限流、验证码、复杂密码策略、账户锁定策略等（**暂不考虑**，后续按需补）。
- **错误提示**：登录失败宜使用 **模糊文案**（如「用户名或密码错误」），避免枚举有效账号。
- **HTTPS**：生产环境必须使用；本地开发可 HTTP，与前端约定即可。

---

## 9. 前端 `didi-Vue` 约定

- **登录**：登录成功后保存 token，请求拦截器统一带 **Bearer**（若项目尚未接登录页，按此约定补齐）。
- **菜单与路由**：与后端返回的菜单树（含 **`path`、`component`**）一致；**`perms`** 与按钮显隐约定一致即可。
- **HTTP 状态**：**401** → 清 token 跳转登录；**403** → 无权限提示（勿与网关 502 混淆）。
