# 订单服务幂等与并发方案说明

## 目标

在 `order` 服务中，用一套可落地的方案解决以下问题：

- 重复提交
- 并发状态变更冲突
- 跨服务最终一致性

核心思路：

- 请求级幂等（`requestId`）
- 数据库条件更新（按状态 CAS）
- 事务型 Outbox
- Kafka 事件分发
- 消费端幂等

## 为什么只用 Kafka 不够

Kafka 适合异步解耦与削峰，但它本身不能直接保证业务幂等和状态机正确性。

如果只依赖 Kafka，常见问题有：

- 仍可能出现重复生产、重复消费
- 分区有序不等于业务并发冲突被正确处理
- 数据库更新成功但消息发送失败，容易造成状态不一致

所以订单状态流转必须先在数据库层做强校验。

## 推荐流程

### 1) 写接口入口（同步链路）

针对写操作（`create/accept/cancel/start/finish`）：

1. 请求必须携带 `requestId`
2. 查询并落幂等记录
3. 执行状态条件更新
   - `update ... where order_no = ? and status = ?`
4. 更新成功后，在同一事务写入 `order_event`
5. 在同一事务写入 outbox 事件
6. 返回结果（可按 `requestId` 缓存响应快照）

### 2) 事件发布（异步链路）

- 后台发布器扫描 outbox 待发布记录并投递到 Kafka
- 仅在发送成功后标记 outbox 为 `PUBLISHED`
- 失败按退避策略重试

### 3) 消费端处理

- 消费前先检查 `processed_event` 中的 `eventId/requestId`
- 已处理：直接跳过
- 未处理：执行业务并原子写入“已处理标记”

## 最小表设计

### `order_idempotent_record`

建议字段：

- `id`
- `request_id`（唯一）
- `action_type`（CREATE/ACCEPT/CANCEL/...）
- `order_no`
- `status`（INIT/SUCCESS/FAILED）
- `response_snapshot`（json/text，可选）
- `created_at`、`updated_at`

用途：

- 防止重复提交/重试造成副作用
- 重复请求返回稳定结果

### `order_outbox_event`

建议字段：

- `id`（事件 ID）
- `event_type`
- `aggregate_type`（`ORDER`）
- `aggregate_id`（`order_no` 或 `order_id`）
- `payload`（json）
- `publish_status`（PENDING/PUBLISHED/FAILED）
- `retry_count`
- `next_retry_at`
- `created_at`、`updated_at`

用途：

- 连接本地事务与 Kafka 发布，保证一致性

### `order_processed_event`

建议字段：

- `id`
- `consumer_group`
- `event_id`（或 `request_id`）
- `processed_at`

唯一键：

- (`consumer_group`, `event_id`)

用途：

- 保证消费端幂等

## 订单状态机并发规则

即使使用 Kafka，状态流转门闩仍应放在订单库：

- 当前状态只允许一个合法后继状态
- 更新必须携带“当前状态条件”（CAS）
- 对非法状态迁移显式拒绝

示例：

- `ASSIGNED -> ACCEPTED`：只能有一个司机抢到
- `STARTED -> FINISHED`：重复完单不能重复计费

## 落地优先级（MVP）

优先级 1：

- 所有写接口强制 `requestId`
- 建 `order_idempotent_record`
- 状态流转使用 CAS 更新

优先级 2：

- 引入事务 Outbox
- 实现 Outbox 发布任务

优先级 3：

- 消费端去重（`order_processed_event`）
- 监控重试堆积并告警

## 风险与控制

- 风险：幂等键使用不规范
  - 控制：在网关/客户端统一 `requestId` 生成规则
- 风险：Outbox 堆积
  - 控制：监控待发布数量和重试滞留时长
- 风险：重复消费
  - 控制：消费端强制唯一去重键
- 风险：跨服务链路延迟
  - 控制：按最终一致性设计补偿策略
