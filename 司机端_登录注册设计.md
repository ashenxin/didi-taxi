# 司机端登录/注册设计（密码 + 短信验证码）

本文档定义 **司机端（司机 App/H5）登录与注册** 的首期设计方案：**手机号+密码** 与 **手机号+短信验证码（OTP）** 两类入口，并引入「司机档案审核」的最小建模，使得 **“能否登录 App” 与 “能否接单” 解耦**。  
范围包含：接口契约、JWT 约定、网关放行、基础风控（频控/次数限制）、MySQL/Redis 设计、错误码与联调要点。  
**本文档仅用于讨论设计，不包含代码实现。**

**与下一期能力的关系**：**派单确认窗口**、**司机端 WebSocket + presence**、**退出登录与在线状态一致** 等目标设计，见 **§4.2/§4.4、§6.6、§8、§12** 及《`乘客司机端_最小闭环接口调用文档.md`》《**`司机端_上线听单与接单设计.md`**》。

---

## 1. 背景与目标

- **背景**
  - 司机侧对外 BFF：`driver-api`，路由前缀 `/driver/**`（经 `gateway` 转发）。
  - 司机档案主数据在 `capacity` 服务（库 `capacity` 表 `driver`），目前无密码字段。
  - 网关已启用 JWT 校验，并按路径做 `aud` 隔离（`/driver/**` 期望 `aud=driver-bff`）。
  - 当前 `driver-api` 多个业务接口由客户端传入 `driverId`（path/query/body），需要与 JWT 身份绑定，避免伪造。

- **目标（MVP）**
  - 支持司机端 **注册**（短信注册、密码注册）与 **登录**（短信登录、密码登录）。
  - 登录/注册成功后返回 **Access Token（JWT）**；后续请求统一走 `Authorization: Bearer <token>`。
  - 网关对白名单接口放行，其余 `/driver/**` 要求 JWT。
  - 验证码具备最小必要的 **发送频控 + 日上限 + TTL + 一次性校验**。
  - **能否登录 App 不限制**（允许被限制接单的司机登录后申诉/补件），**接单相关动作必须严格校验司机接单资格与审核状态**。

---

## 2. 术语

- **Core**：`capacity` 服务（司机档案与接单资格主数据）。
- **BFF**：`driver-api` 服务（司机端聚合层，对外暴露 `/driver/api/v1/**`）。
- **Gateway**：`gateway` 服务（统一入口，端口 8080，路由转发 `/driver/**` 至 BFF）。
- **OTP**：一次性验证码（短信验证码）。
- **审核流水（Audit Record）**：司机资料审核的每一次处理记录（通过/驳回/需补件等）。

## 3. 数据模型（MySQL）

### 3.1 `capacity.driver`（现有表，需扩展）

现有 `driver` 关键字段（节选）：

- `id`：司机ID
- `phone`：手机号
- `id_card` / `license_*` 等：实名与资质材料字段
- `monitor_status`：听单状态
- `can_accept_order`：是否可接单（业务开关/限制接单）
- `rpt_status`：上报账号状态（监管/上报维度，见 9.3）

**新增/调整建议（MVP）**

1. **密码摘要**
   - `password_hash` VARCHAR(128) NULL  
   - 语义与乘客侧 `customer.password_hash` 对齐，采用 BCrypt 摘要。
   - 允许为空：表示未设置密码（可仅短信注册/登录）。

2. **审核状态快照（用于“接单拦截”，不用于“能否登录”）**
   - `audit_status` INT NOT NULL DEFAULT 0  
   - 枚举建议（可后续调整）：
     - `0` 待提交/待完善（资料不全）
     - `1` 审核中
     - `2` 审核通过（可开通接单前置条件之一）
     - `3` 审核驳回/需补件（限制接单，但允许登录）
     - `4` 暂停/冻结接单（允许登录，需申诉/处理）
   - `audit_updated_at` DATETIME（可选）：状态更新时间
   - `audit_last_record_id` BIGINT（可选）：指向最新审核记录，便于快速定位原因

3. **手机号唯一（可选但强烈建议，兼容导入与自助注册）**
   - 同乘客方案：`phone_active` 生成列 + 唯一索引，保证 `is_deleted=0` 时手机号不重复。
   - 若暂不改，可先靠应用层兜底；但在“自助注册 + 批量导入”并存后，建议补上数据库约束。

> 说明：本方案不改变现有 `can_accept_order` 语义；接单前置判断应综合 `audit_status` 与 `can_accept_order`。

### 3.2 新表：`driver_audit_record`（审核流水表）

每次审核都插入一条记录，承载“通过/驳回/需补件”的详细原因。司机主表仅保留当前状态快照。

字段建议：

- `id` BIGINT PK
- `driver_id` BIGINT NOT NULL
- `result_status` INT NOT NULL（同 `audit_status` 的结果口径，或更细的结果枚举）
- `reason` VARCHAR(2000) NULL（驳回原因/补件要求；也可改 JSON 承载结构化原因码）
- `operator_type` INT NOT NULL（0 系统、1 运营、2 客服、3 审核员等）
- `operator_id` BIGINT NULL
- `submission_id` BIGINT NULL（强烈建议：关联一次资料提交/版本；没有版本也可以先空）
- `created_at` DATETIME DEFAULT CURRENT_TIMESTAMP

查询约定：

- 司机端展示“当前原因/要求补什么”：读取 `driver.audit_last_record_id` 指向的记录；或按 `driver_id` 倒序取最新一条。

### 3.3 批量导入与自助注册的边界

- **批量导入**：写入 `driver_source=1`，可选择是否写入 `password_hash`（例如初始密码/待设密），但不应绕过审核流程；导入完成后仍可能处于 `audit_status` 非通过状态。
- **自助注册**：写入 `driver_source=0`，初期允许“仅手机号建档 + 短信验证”，但接单必须满足审核通过。

## 4. JWT 约定（司机端）

### 4.1 Token 类型

- **Access Token**：JWT（HMAC 签名），由 `driver-api`（BFF）签发，`gateway` 校验。

### 4.2 Claims 约定

- **`sub`**：司机 ID（`driver.id`，字符串）
- **`exp`**：过期时间（必须）
- **`iat`**：签发时间（建议）
- **`aud`**：**必须写入** `driver-bff`（与网关 `gateway.jwt.audience-driver` 对齐）
- **`token_version`**（目标，已拍板）：非负整数，**每成功登录递增**；与 Redis 中当前版本一致时 token 有效（见 **§4.4**）。
- （可选）`phone`：手机号（便于前端展示/排障；不作为授权依据）

### 4.3 密钥与过期时间

- **签名密钥（结构隔离）**：司机端使用 **`JWT_SECRET_DRIVER`**（`app.jwt.secret`），网关验签使用 `gateway.jwt.secret-driver`，两者必须一致；建议与 admin/app 端密钥不同以防跨端 token 复用。
- **自然过期时间**：**8 小时**（`app.jwt.expiration-seconds = 28800`）；JWT **`exp`** 与登录响应 **`expiresIn`（秒）** 与此一致。
- **提前失效**：**`token_version` 变更或登出** 可在 `exp` 到达前使 Token **401**，与自然过期正交。

### 4.4 全局唯一会话（`token_version` + Redis，已拍板）

- **目标**：同一手机号（同一 `driver_id`）**全局仅一个有效 JWT**；新登录或 **登出** 使用旧 token 的请求 **401**。
- **实现要点**：
  - Redis 存当前 **`token_version`**；**实现键名**：**`driver:tv:{driverId}`**（`driver-api` `DriverTokenVersionStore`；与设计稿中的 `driver:token:ver:{driverId}` 同义）。登录成功 **INCR**；**登出**同样 **INCR**，使旧 JWT 内嵌版本失效。
  - JWT 内携带 **`tv`** claim（与 Redis 当前值一致）；**`driver-api`** 在校验过滤器中比对 **`tv`** 与 Redis（网关仅验签与 `aud`）。
- **与 WebSocket**：旧 token 失效后，**旧 WebSocket** 应被关闭或握手失败；新连接使用新 token。
- **详情**：业务规则与调度侧衔接见《`司机端_上线听单与接单设计.md`》**§4**。

## 5. 网关放行（Public 白名单）

司机端未登录接口必须放行，否则前端无法获得 token。

建议白名单（`POST`，5 个）：

- `/driver/api/v1/auth/sms/send`
- `/driver/api/v1/auth/register-sms`
- `/driver/api/v1/auth/register-password`
- `/driver/api/v1/auth/login-sms`
- `/driver/api/v1/auth/login-password`

其他 `/driver/**` 均要求 `Authorization: Bearer <token>`。

同时保持网关 `aud` 校验开启（`gateway.jwt.audience-check-enabled=true`）作为防御纵深，以防止跨端误用 token：

- `/driver/**` 仅接受 `aud=driver-bff`
- `/app/**` 仅接受 `aud=app-bff`
- `/admin/**` 仅接受 `aud=admin-bff`

## 6. API 设计（对外：driver-api / BFF）

统一前缀：`/driver/api/v1/auth`

统一响应：`{ code, msg, data }`（与现有工程风格一致）。

### 6.1 发送短信验证码

- **POST** `/driver/api/v1/auth/sms/send`

请求：

```json
{ "phone": "13900000000" }
```

成功：

```json
{ "code": 200, "msg": "success", "data": null }
```

失败：

- `400`：手机号格式非法
- `429`：发送过于频繁 / 今日次数已达上限

### 6.2 短信注册

- **POST** `/driver/api/v1/auth/register-sms`

请求：

```json
{ "phone": "13900000000", "code": "123456" }
```

成功响应：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "accessToken": "<jwt>",
    "tokenType": "Bearer",
    "expiresIn": 28800,
    "driver": {
      "id": 80001,
      "phone": "13900000000",
      "auditStatus": 0
    }
  }
}
```

行为规则：

- OTP 校验成功后 **一次性删除**（防复用）
- 若手机号已存在：
  - 若为已删除记录：按“重新激活”策略（实现时明确）
  - 若为未删记录：按业务选择
    - 方案 A（推荐）：当作登录处理（返回 token），避免“重复注册”分支复杂
    - 方案 B：返回 `409` 提示已注册请登录

### 6.3 密码注册（短信校验 + 设置密码）

> 仅 `phone+password` 不能证明手机号归属，因此密码注册必须绑定短信验证码。

- **POST** `/driver/api/v1/auth/register-password`

请求：

```json
{ "phone": "13900000000", "code": "123456", "password": "plaintext" }
```

成功响应同 6.2。

失败：

- `401`：验证码错误或过期
- `409`：手机号已被占用（若采用方案 B）

### 6.4 短信登录

- **POST** `/driver/api/v1/auth/login-sms`

请求：

```json
{ "phone": "13900000000", "code": "123456" }
```

行为规则：

- OTP 校验成功后一次性删除
- 若手机号不存在：可选择
  - 方案 A（与“注册”区分清晰）：返回 `404/401` 提示先注册
  - 方案 B（降低流程摩擦）：视同注册（创建 driver 档案最小行）

> 本文档推荐 **方案 A**：司机业务更强合规属性，注册可承载更多提示/协议/引导；登录不隐式创建司机。

### 6.5 密码登录

- **POST** `/driver/api/v1/auth/login-password`

请求：

```json
{ "phone": "13900000000", "password": "plaintext" }
```

失败：

- `401`：手机号或密码错误

行为规则：

- 若 `password_hash` 为空：`401`，提示“未设置密码，请使用验证码登录或先设置密码”

### 6.6 退出登录（已实现服务端作废 + 客户端协同）

1. **接口**：**`POST /driver/api/v1/auth/logout`**（**`driver-api`**；需 **`Authorization: Bearer`**，**不**放入网关 Public 白名单）。详见《`司机端_登录注册接口文档.md`》**§9**。
2. **服务端语义（已落地）**：对 Redis **`driver:tv:{driverId}`** 执行 **INCR**，使 JWT 内 **`tv`** 与 Redis 不一致，旧 token 在 **`driver-api`** 侧 **401**。与 **§4.4** 一致。
3. **客户端必做**：删除本地 **`accessToken`**；若已建 **WebSocket** 则断开并停止心跳。
4. **目标协同（与 WebSocket / presence / 派单，分期落地）**：**presence 离线**、**听单关闭**、**待确认 offer 不再推给该司机** 等仍以《`司机端_上线听单与接单设计.md`》与订单/调度实现为准；HTTP 登出仅保证 **JWT 会话作废**。
5. **Redis 司机池（规划，与 JWT 作废正交）**：**下线听单**或 **登出** 时，须从 **Redis 司机池** 移除该 `driverId`，避免仍被匹配派单；**listAssigned 是否仍展示已指派单**见《`乘客司机端_Redis与听单下线策略.md`》**§4（策略 A）**。
6. **幂等**：多次登出均为再次 INCR；旧 token 首次成功后即失效。

## 7. API 设计（对内：capacity / Core）

建议 BFF 不直接访问数据库，统一通过 `capacity` 服务完成：

- `sms/send`（写 OTP + 频控 + 发送）
- `register-sms`（验 OTP + 创建司机档案最小行）
- `register-password`（验 OTP + 创建档案 + 写 password_hash）
- `login-sms`（验 OTP + 返回司机简要信息）
- `login-password`（验密 + 返回司机简要信息）

Core 返回给 BFF 的数据应尽量小：`driverId/phone/auditStatus/canAcceptOrder/monitorStatus` 等；由 BFF 负责签 JWT。

## 8. Redis 设计（OTP 与风控）

司机端 OTP 与风控建议独立前缀（不要复用乘客的 `app:*`），避免同手机号跨端相互影响。

### 8.1 Key 约定（建议）

- **OTP**：`driver:otp:{phone}` → `code`（TTL = 300s）
- **短信发送最小间隔**：`driver:sms:gap:{phone}` → `1`（TTL = 60s）
- **短信发送日计数**：`driver:sms:daily:{yyyyMMdd}:{phone}` → `count`（TTL = 2d）
- **登录失败日计数（密码+短信合并）**：`driver:login:fail:{yyyyMMdd}:{phone}` → `count`（TTL = 2d）
- **当日登录封禁**：`driver:login:ban:{yyyyMMdd}:{phone}` → `1`（TTL = 2d）
- **`token_version`（已拍板）**：实现为 **`driver:tv:{driverId}`** → 当前版本号（**无 TTL 或长 TTL**，以业务为准）；与 **§4.4**、**§6.6** 一致（设计稿曾写 `driver:token:ver:*`，与现网键名等价）。
- **WS `messageId` 去重（已拍板）**：如 `driver:ws:msgid:{messageId}` → `1`（**TTL = 8 小时**），防重复处理；键名实现时可调整。

### 8.2 频控与次数建议（MVP，默认与乘客端一致）

- **发送间隔**：60 秒
- **日上限**：5 次 / 手机号 / 自然日
- **登录失败上限**：5 次 / 手机号 / 自然日（密码与短信失败合并）；达到后当日禁止登录

> 后续可按司机端风控更严格（例如更低上限或加入 IP/设备维度）。

## 9. 关键鉴权规则（对订单/接单链路的影响）

### 9.1 身份绑定：`driverId` 必须来自 JWT（`sub`）

网关校验 token 后会注入 `X-User-Id = <sub>`。司机端业务接口必须遵循：

- 对接单/状态流转等写操作：**以 `X-User-Id` 为准**，忽略客户端 `driverId`，或强校验二者一致（否则拒绝）。
- 目的：防止客户端伪造 `driverId` 操作他人的订单。

### 9.2 接单拦截：审核状态与可接单开关

司机 **HTTP 登录**不受审核状态限制（可登录后查看原因/补件）。以下动作必须拦截：

- 上线听单（`monitor_status` 相关）
- 接单/到达/开始/完单等订单状态推进

**WebSocket（已拍板）**：**审核未通过不允许建立连接**（握手拒绝），业务规则最简单；详见《`司机端_上线听单与接单设计.md`》**§5.3**。

判断建议（示例）：

- `audit_status == APPROVED(2)` 且 `can_accept_order == 1` 才允许 **听单/接单** 等
- 否则 **HTTP 业务接口** 返回 `403`，并提示引导至资料完善/申诉入口（前端依据 `audit_status` 展示“原因/补件要求”）

### 9.3 `rpt_status` 的使用建议（监管/上报维度）

`rpt_status` 建议作为“监管/上报账号有效性”的维度，而不是直接等同于“能否登录/能否接单”：

- 若监管要求上报失效必须停运：可在接单校验中加一条 `rpt_status==0` 的硬条件
- 若只是提示整改：可仅在后台/司机端展示，并通过 `audit_status` 或 `can_accept_order` 来控制接单

## 10. 错误码与文案（建议）

> 文案可与乘客端保持一致风格。

- `400`：参数非法（手机号格式、验证码格式等）
- `401`：登录失败（验证码错误/过期、手机号或密码错误、未设置密码）
- `403`：禁止接单（审核未通过/需补件/暂停接单等）  
  - **注意**：`403` 不用于“禁止登录”（本方案允许登录）
- `409`：手机号已注册（若注册接口采用“强区分注册/登录”策略）
- `429`：发送过于频繁 / 今日次数已达上限 / 登录失败次数过多（当日封禁）
- `502`：下游服务不可用

---

## 11. 联调要点（最小步骤）

1. 司机端发码：`POST /driver/api/v1/auth/sms/send`
2. 注册（建议先走短信注册）：`POST /driver/api/v1/auth/register-sms`
3. 拿到 token 后，请求 `/driver/**` 业务接口时带：
   - `Authorization: Bearer <token>`
4. 接单/上线类接口若返回 `403`：
   - 前端展示 `audit_status` 与最新审核记录（`driver_audit_record` 最新一条）的 `reason`

## 12. WebSocket、presence 与派单确认（目标架构，未实现）

> **详细已迁移至《`司机端_上线听单与接单设计.md`》**；本节保留索引与衔接说明。

### 12.1 为什么需要 WebSocket + presence

- **HTTP 轮询**难以保证司机 **真实在线**，不利于 **派单确认窗口**（短时内必须让司机看到并操作）。
- **Presence**（在线/可交互）应由 **服务端** 根据 **连接状态 + 应用层心跳** 维护；进程被杀、断网、登出均应反映为离线。

### 12.2 职责划分（建议）

| 能力 | 建议载体 | 说明 |
|------|----------|------|
| 登录注册 / 业务 REST | HTTP + JWT | 与现网一致 |
| 心跳、在线、待确认派单推送 | **WebSocket** | 连接建立后绑定 `driverId`；服务端下发 **`offerExpiresAt`** |
| 听单开关、资质校验 | `capacity` + BFF | `monitor_status`、`can_accept_order`、`audit_status` 等 |
| **确认超时** | **服务端定时任务 / 订单状态机** | **禁止**以客户端倒计时为唯一裁决 |

### 12.3 退出登录与断线

- **主动登出**：见 **§6.6**，必须 **关 WS + 清 presence**。
- **断线**：超过 **N 次未收到心跳** 或 **超过 T 秒** 无有效消息，服务端标记离线；与 **派单确认** 结合时，未确认订单按 **§1.1a（闭环文档）** 规则释放。

### 12.4 与网关的关系

- WebSocket **走 `gateway:8080`**（与 HTTP 同入口）；**须支持 WebSocket 升级**；多实例时 **Sticky** 或 **Redis 等共享推送**，见《`网关服务_技术.md`》补充说明与《`司机端_上线听单与接单设计.md`》**§5.2**。

### 12.5 派单确认窗口（产品语义，与订单域一致）

- **10s**、**超时先回 `ASSIGNED`**、**调度 5s**、**order_event** 等：**《`司机端_上线听单与接单设计.md`》**。

---

*若仅讨论登录注册而不涉及实时派单，可忽略 §12；讨论接单改造时以 **《`司机端_上线听单与接单设计.md`》** 为主。*

**名词入门（WS URL、握手鉴权、消息 schema、订单状态字段等）**：见《`司机端_WebSocket与实时协议入门.md`》。
