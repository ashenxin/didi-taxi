# 后台管理系统：计价管理设计

本文档描述 **计价域**（运力公司维度的计价规则 `fare_rule`）的产品与技术共识，与《后台管理系统_计价管理_接口文档.md》配套。  
全站登录与省/市数据域见《后台管理系统_权限清单与鉴权设计.md》及《后台管理系统_权限与接口文档.md》**§4.7**。运力公司档案见《后台管理系统_运力配置_设计.md》。

| 包含 | 不包含 |

| `calculate-service` 中 `fare_rule` 领域含义与管理端约束 | 网关、`sys_*` 账号与菜单详设（见权限文档） |
| `admin-api` 下 **`/admin/api/v1/pricing/fare-rules`** BFF | **乘客端** `passenger-api` 下单链路与 **`/api/v1/calculate/estimate`** 的详细编排（接口文档仅摘要） |
| 计价规则与 **`capacity.company`** 绑定（`company_id` / `company_no`） | 独立「定价引擎」多版本灰度、动态计价公式扩展（以当前表结构为准） |

---

## 2. 架构边界

- **`calculate-service`**：库 **`calculate`**，表 **`fare_rule`**；提供规则 **CRUD** 与 **费用预估**（`POST /api/v1/calculate/estimate`）。HTTP 前缀见接口文档。
- **`admin-api`**：前缀 **`/admin/api/v1/pricing/fare-rules`**；**Feign → calculate**；**不直连计费库**。列表中的 **`companyName`** 由 BFF 调 **`CapacityClient#companyDetail`** 填充。
- **前台 `didi-Vue`**：路由 **`/pricing/fare-rules`**（菜单以 `sys_menu` 为准）；新建/编辑页从 **运力公司分页** 拉取可选公司，选公司后 **自动带出** 该公司档案中的 **省、市编码**，与规则行上的 **`province_code` / `city_code`** 保持一致。

---

## 3. 领域模型（与实现对齐）

- **计价规则（FareRule）**：一行表示在某 **运力公司**、某 **行政省/市（编码）**、某 **产品线（`product_code`，如 ECONOMY / COMFORT）** 下生效的一套 **分段计价参数**，并带 **`effective_from` / `effective_to`** 有效期；**`effective_to` 为空** 表示当前仍有效。
- **与运力公司的关系**：`fare_rule.company_id` 指向 **`capacity.company.id`**；`fare_rule.company_no` 与 **`capacity.company.company_no`** 一致，便于审计与展示。规则上的 **省、市** 必须与该公司档案中的 **省、市** 一致（BFF 保存前校验）。
- **产品线**：同一公司、同一省市下，**不同 `product_code` 可各配置一条（在时间不重叠的前提下）**；即业务唯一维度为 **公司 + 省 + 市 + 产品线**。
- **有效期重叠**：在同一上述四维下，**未逻辑删除**（`is_deleted = 0`）的多条规则，其 **`[effective_from, effective_to]`**（空结束视为无穷远）**不得时间重叠**，否则创建/更新由 **`calculate-service`** 拒绝。允许存在 **已结束** 的历史版本与 **当前有效** 版本共存（时间不重叠）。

### 3.1 管理端写操作（新增 / 编辑 / 删除）

- **新增 / 编辑**：必选 **运力公司**；省、市经 **`AdminDataScope#scopeFareRuleWrite`** 与登录域对齐后，BFF **`validateCompanyForFareRule`** 拉取公司详情，校验 **省、市与公司一致** 并 **写入 `company_no`** 再转发 calculate。
- **删除**：**逻辑删除**（`is_deleted != 0`），由 calculate 实现。

### 3.2 计费公式（概要）

- 单笔预估：**起步价**；超出 **起步含里程 / 含时长** 部分按 **每公里 / 每分钟** 加价；可选 **最低消费**、**封顶价**。具体计算以 **`calculate-service`** `EstimateController` 为准。

---

## 4. 数据权限

- 与运力、订单域相同：**`AdminDataScope`** 对列表查询合并 **省、市**，对读写 body 锁定 **省、市**；跨域资源 **404**（掩蔽），筛选项越权 **403**。详见《权限与接口文档》**§4.7**。

---

## 5. 与下单链路的简要关系（非管理端详设）

- 乘客「创建并指派」时，需在已知 **承运运力公司** 的前提下对 **`fare_rule`** 做预估；实现上通常 **先得到候选司机所属 `company_id`**，再在 **`estimate` 请求中带 `companyId`** 命中规则。

---

## 6. 风险与后续

- **运营配置**：某公司某产品线若未配置有效规则，下游估价可能失败或返回业务 **404**（具体以 `passenger-api` 与 calculate 约定为准）。
- **一致性**：公司 **删改** 与历史 **`fare_rule`** 行的外键关系为 **弱关联**（仅存 id/no）；重大变更需运营或脚本同步规则。
