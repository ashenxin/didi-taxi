# 第一期 MVP：乘客派单司机闭环（PRD）

> 本 PRD 用“可落地验收”的方式描述第一期要交付的乘客/司机打车闭环：每条关键规则都能被前端实现、被测试验收、被用户理解。

---

## 0. 文档说明

- **目标读者**：产品 / 前端 / 后端 / 测试
- **文档定位**：本期的“产品口径 + 验收口径”。接口字段与技术实现细节不在本文展开（后续落到 API/TECH 文档）。

---

## 1. 一期范围与非目标

### 1.1 本期必须交付（In Scope）

- **乘客端**：登录、下单、等待派单/跟踪、订单详情、取消订单、看到取消方与取消原因（系统/乘客）。
- **司机端**：登录、上线/下线听单、指派列表、接单、拒单（新增按钮）、已接单后取消、行程推进（若已支持到达/开始/完单则一并验收）。
- **退出登录（本期新增）**：乘客/司机均支持退出登录；**在司机到达前**退出登录会导致本单按“取消”处理（见 §5.6）。
- **系统兜底**：从下单成功开始累计等待 180 秒（3 分钟）仍未形成“司机已接单” → 系统取消，并给用户明确原因。

### 1.2 本期不承诺（Out of Scope）

- 乘客端 WebSocket 推送与实时在线态裁决。
- 司机端业务 WebSocket + presence。
- 取消费/违约金/补偿结算等费用规则（本期不讨论、不验收）。
- 两段式 Outbox + Kafka 异步派单作为默认主链路（可作为后续演进方案存在，但不作为本期验收点）。

## 2. 统一口径（本期铁律）

### 2.1 “等待态”统一定义

乘客侧以下阶段都属于同一件事：**我在等车**（无需用户额外操作）：

- 正在派单（未分配到司机）
- 等待司机接单（已派给某司机，但司机尚未接单）
- 重新派单/改派中（司机拒单、超时未接、已接单后取消、互斥释放改派等导致系统重新找车）

乘客端可统一展示为“等待态”，必要时用副标题区分“正在重新派单”。

### 2.2 总体等待 180 秒兜底（不重置）

- **计时起点**：乘客下单成功（订单创建成功）时刻。
- **计时终点**（满足任一项即结束）：
  - **成功**：订单进入“司机已接单”
  - **用户退出**：乘客取消成功
  - **失败兜底**：累计等待达到 **180 秒（≈3 分钟）**仍未接单 → **系统取消**
- **关键点**：中间发生“派到了 → 又取消/拒单/收回 → 重新派单”不会重置 180 秒。

### 2.3 兜底结果（系统取消）

当累计等待达到 180 秒仍未形成“司机已接单”：

- 订单进入终态：**已取消（系统取消）**
- 对外原因语义：**附近暂无可用车辆 / 当前暂无司机接单**
- 推荐短文案：`附近暂无可用车辆，请稍后重试`

### 2.4 取消权（乘客）

- 在所有等待态（包括重新派单/改派中）里，**乘客始终有权取消**。
- 乘客取消成功后为终态：不应再继续派单/改派，不应再被系统兜底取消覆盖。

### 2.5 原因采集（轻量、无输入、不对端展示）

- **不要求**乘客/司机输入长文本原因（不做自由输入框）。
- 原因以**前端写死的单选项**为主（最小可用）。
- **不把原因展示给对方**：
  - 乘客不需要看到司机拒单/取消原因；
  - 司机也不需要看到乘客取消原因。

### 2.6 “退出登录 = 取消”（仅限司机到达前）

- **规则**：乘客/司机发起退出登录时，若当前存在“司机到达之前”的进行中订单，则视为“取消本单”。
- **限制**：司机已到达后，再退出登录不应再触发“取消本单”（此时应提示用户先完成行程/按规则处理，不在本期展开费用与风控）。
- **目的**：用户退出登录时不需要再额外思考“要不要先取消订单”，系统以最直观的方式收敛。

### 2.7 订单状态说明（中文口径，供联调/验收对照）

> 说明：以下是 order-service 内部状态的业务含义。**乘客端展示**可按本 PRD 的“等待态”做归并（不要求把每个内部状态都暴露给用户）。

- **CREATED（待派单）**：订单已创建，系统正在找车，尚未分配司机（乘客侧通常展示“正在为你寻找车辆”）。
- **ASSIGNED（已派单）**：已分配司机（`driver_id` 已写入），但司机可能尚未接单（乘客侧可展示“已派单/等待司机接单”或仍归并为等待态）。
- **PENDING_DRIVER_CONFIRM（待司机确认）**：派单确认窗口内，等待司机在截止时间前确认接单；超时会回到“已派单/等待下一轮”。
- **ACCEPTED（司机已接单）**：司机确认接单成功，订单从“等待”进入“已接单”。
- **ARRIVED（司机已到达上车点）**：司机到达乘客上车点。**本期口径：到达后不允许再通过“退出登录/取消”来结束本单**（见 §5.6 与 E2E-06）。
- **STARTED（行程已开始）**：行程开始（乘客已上车），进入行程中阶段。
- **FINISHED（已完单）**：行程结束，订单完成。
- **CANCELLED（已取消，终态）**：订单已取消。取消方与原因需通过 `cancel_by` / `cancel_reason`（或等价字段）区分（乘客取消/系统取消等）。

## 3. 乘客端页面清单与验收点（一期）

> 页面可以是独立路由，也可以是同一页的不同模块；验收以“用户能看到并理解状态、能完成操作”为准。

### 3.1 登录页

- **验收点**：
  - 登录成功后可下单与查看订单
  - 未登录访问业务页会提示未授权并引导登录（或等价行为）
  - 支持退出登录（见 §5.6），退出后业务页不可访问或需重新登录

### 3.2 下单页（发起叫车）

- **最小要素**：起点/终点、`立即下单` 按钮（产品类型/车型若一期固定可隐藏）
- **关键交互**：
  - 点击下单成功 → 自动进入“等待派单/订单跟踪”
  - 若已有进行中订单：前端可提示/禁用；服务端冲突（如 409）为最终兜底
- **验收点**：
  - 正常下单：能进入跟踪流程
  - 重复下单：有明确提示且不会产生多笔进行中订单

### 3.3 等待派单页 / 订单跟踪页（等待态主页面）

- **页面文案**：
  - 主标题：`正在为你寻找车辆`
  - 发生改派时副标题：`正在重新派单，请稍候`
- **按钮**：`取消订单`（必须）
- **验收点**：
  - 司机拒单/超时未接/司机取消/互斥释放后：乘客看到“正在重新派单”（无需操作）
  - 任意等待态可取消成功（终态稳定）
  - 累计到 180 秒仍未接单：系统取消（原因语义正确）

### 3.4 订单详情页

- **最小展示**：
  - 当前状态（等待/已派单/司机已接单/已取消/已完成）
  - 司机信息（已派单或已接单时）
  - 取消信息：取消方 + 取消原因（已取消时必须可见）
- **验收点**：
  - 刷新订单详情能看到状态推进
  - 系统取消与乘客取消可区分（至少在取消方/原因处体现）
  - 终态稳定：已取消后不会再变回已派单/已接单

## 4. 司机端页面清单与验收点（一期）

### 4.1 登录页

- **验收点**：登录成功后可进入听单/指派列表；未登录访问业务页被拦截或提示
  - 支持退出登录（见 §5.6），退出后业务页不可访问或需重新登录

### 4.2 上线听单页 / 听单开关

- **要素**：听单开关 `上线听单/下线`
- **验收点**：
  - 上线后可被派单（环境满足时）
  - 下线后不应收到新指派（已指派订单的可见性按实现一致即可）

### 4.3 指派订单列表页（派给我的订单）

- **卡片按钮**：
  - `接单`
  - `拒单`（本期新增，必须）
- **状态提示**：
  - 超时未接被收回：`已被系统收回（超时未接单）`（Toast 或置灰提示）
  - 拒单成功：`已拒绝，订单已收回`
- **验收点**：
  - 接单成功后进入行程/订单详情可推进
  - 拒单后该单不可再接；乘客进入“正在重新派单”
  - 不操作直到超时：司机看到“已被系统收回”；乘客进入“正在重新派单”

### 4.4 拒单弹窗（轻量原因选择，无输入框）

- **交互**：点击拒单 → 单选原因 → 确认
- **原因选项（前端写死即可）**：
  - 距离太远 / 临时有事 / 车辆问题 / 其它
- **验收点**：
  - 无自由输入框
  - 拒单原因不展示给乘客

### 4.5 订单详情/行程页（已接单后）

- **按钮**：
  - 行程推进（到达/开始/完单，若一期支持）
  - `取消订单`（已接单后取消，本期必须）
- **取消交互**：
  - 单选原因（前端写死，无输入框）
  - 成功提示：`已取消，系统将重新派单`
- **验收点**：
  - 司机取消后：乘客进入“正在重新派单”
  - 司机取消后：该单不可再推进到达/开始/完单

## 5. 关键业务规则（可验收条款）

### 5.1 乘客主动取消（等待态）

- **规则**：等待态（含重新派单/改派中）乘客可随时取消
- **结果**：已取消（乘客取消），终态稳定
- **验收**：等待态点击取消 → 详情可见乘客取消；之后不会再派到司机

### 5.2 司机接单互斥（只允许成功接一单）

- **规则**：同一司机同一时间只能成功接一单
- **其余单处理**：释放改派 → 乘客进入“正在重新派单”（不展示系统取消）
- **验收**：同司机两单待接 → 接成一单 → 另一乘客进入“重新派单”

### 5.3 司机拒单

- **规则**：司机可在未接单阶段点击拒单；拒单后该单对该司机立即失效
- **乘客侧**：进入“正在重新派单”（无需操作）
- **验收**：拒单后司机不可再接；乘客进入重新派单；总等待不重置

### 5.4 司机已接单后取消（行程开始前）

- **规则**：司机有权取消已接订单（至少覆盖行程开始前）
- **乘客侧**：进入“正在重新派单”（无需操作）
- **验收**：司机取消后乘客进入重新派单；司机侧不可再推进行程

### 5.5 司机超时未接 → 系统收回

- **规则**：司机不操作 → 到期系统收回
- **司机侧**：需提示“已被系统收回”
- **乘客侧**：进入“正在重新派单”
- **验收**：超时后司机看到提示；乘客进入重新派单

### 5.6 退出登录（Logout）与订单取消（司机到达前）

- **规则**：乘客/司机调用“退出登录”：
  - 若当前存在进行中订单，且仍处于**司机到达前**（未到达上车点）阶段 → 本单按“取消”处理，订单进入终态（取消方为发起退出的一侧，原因语义为“用户退出登录导致取消”即可）。
  - 若已处于**司机到达后**阶段 → 退出登录不应再触发“取消本单”；应给出明确提示（例如“司机已到达，无法通过退出登录取消订单”）。
- **验收点**：
  - 到达前退出登录：订单最终为已取消，且终态稳定
  - 到达后退出登录：订单不被取消，且给出清晰提示

## 6. 最小验收用例集（E2E）

### E2E-01 正常闭环

- **步骤**：乘客下单 → 司机上线 → 司机接单 →（到达/开始/完单，若支持）
- **预期**：乘客可跟踪状态推进；最终完成或进入期望终态

### E2E-02 无人接单 → 180 秒兜底系统取消

- **步骤**：制造无人可接环境 → 乘客下单等待
- **预期**：累计 180 秒仍未接单 → 系统取消 + “暂无车辆/无人接单”原因语义正确

### E2E-03 拒单/超时未接/司机取消 → 重新派单（无需乘客操作）

- **步骤**：分别覆盖三种触发：
  - 司机拒单
  - 司机不操作超时被收回
  - 司机已接单后取消
- **预期**：乘客进入“正在重新派单”；总等待仍累计；最终成功接单或走兜底系统取消

### E2E-04 等待态乘客取消

- **步骤**：任意等待态（含重新派单）点击取消
- **预期**：乘客取消终态稳定；不再派单；不会被 180 秒兜底覆盖

### E2E-05 退出登录触发取消（司机到达前）

- **步骤**：
  - 场景 A：乘客下单（等待态或已派单/已接单但未到达）→ 乘客退出登录
  - 场景 B：司机接单后（未到达）→ 司机退出登录
- **预期**：
  - 到达前退出登录会导致订单进入取消终态（并可在详情看到取消方/原因语义）
  - 退出后需重新登录才可继续访问业务页

### E2E-06 司机到达后禁止“通过退出登录取消”

- **步骤**：司机接单并进入“已到达”后 → 乘客或司机尝试退出登录以取消
- **预期**：
  - 订单不会因退出登录而被取消
  - 用户获得明确提示“司机已到达，无法取消/无法通过退出登录取消”

## 7. 常见问题与排障口径（给产品/测试/联调）

- **一直在“重新派单”**：表示目前没有司机接单/司机拒单/司机取消/超时收回导致系统持续找车；累计 180 秒未接单会自动系统取消。
- **为什么突然取消了**：若等待累计达到 180 秒仍未接单 → 系统取消（暂无车辆/无人接单）；若用户点了取消 → 乘客取消。
- **司机端为什么单子消失了**：可能是你拒单了 / 超时未接被收回 / 你已接其他订单导致互斥收敛。
