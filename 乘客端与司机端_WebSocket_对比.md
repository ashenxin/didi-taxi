# 乘客端 vs 司机端 WebSocket — 对比稿（规划对齐用）

> 目的：乘客侧准备上 WS 时，与**已落地的司机端 WS**（`driver-api`）在架构、鉴权、消息形态上**对齐差异、复用经验**，减少两套协议并行带来的运维与排障成本。  
> 司机端实现口径还可对照：《`司机端_上线听单与接单设计.md`》《`司机端_WebSocket与实时协议入门.md`》（若与代码不一致以代码为准）。

---

## 0. 乘客端 WebSocket — 已定稿结论（2026-04-30）

> 下列为产品/架构讨论结论，**实现仍以代码为准**；落地后可将本节「规划」改为「现状」并回补类名与配置键名。

### 0.1 推送与数据权威

- **阶段 1（轻量）**：下行只带 **`orderNo` + 单调序号 `seq`**（及与司机端一致的 **`type` / `ts` / `data` envelope**）；客户端据 `seq` 去重/抗乱序后，再 **`GET` 订单详情**；**业务展示以详情 HTTP 为准**，WS 不替代裁决。
- **多单并行**：本期**不考虑**；后续若需要，仍可在消息内带 `orderNo` 扩展，无需推翻本方案。

### 0.2 鉴权与单会话

- **与司机对称**：HTTP 业务 JWT（`audit=1`）→ **`POST /app/api/v1/auth/ws-token`**（路径以最终实现为准）换取 **WebSocket 握手专用 JWT（`audit=2`）**；**`tv` 与 HTTP 一致**，登出/顶号递增 `tv` 后旧 Token 与小票均失效。
- **乘客侧必须具备 `tv`**：与「**不允许两处同时有效登录**」一致；WS 握手须校验 `tv`。
- **同一乘客仅保留一条 WebSocket**：新连接建立时**关闭或顶替**同 `passengerId` 旧会话（实现细节由开发收口）。

### 0.3 连接生命周期（与「无单不长连」）

- **仅在需要跟单的页面**（如下单后进入订单/行程跟单页）建立 WS；**返回首页且本单不再跟实时状态**时**断开**。
- **无进行中、需实时跟进的订单时，不保持 WS 长连接**。

### 0.4 网关与部署

- **与司机侧一致**：生产/联调主干为 **浏览器 → 网关（WebSocket Upgrade）→ `passenger-api`**，`Path=/app/**` 与现有 REST 同域同源（参见《`网关服务_技术.md`》**§3.1**）。
- **本地排障**：可直连 `passenger-api:8100`，环境变量类比司机 `VITE_PASSENGER_WS_BASE_URL`（仅 dev）。
- **`GATEWAY_JWT_REQUIRE_AUTH=true`（默认）时必备**：网关 `JwtAuthenticationGlobalFilter` 须将 **`GET /app/ws/**`** 与 **`GET /driver/ws/**`** 一样列入**握手白名单**（仅 Query `token=`、无 `Authorization`）；否则握手 **401**，浏览器端表现为 **WS 永连不上、狂刷 `ws-token`、仅 2s 轮询兜底**。实现见 `gateway/.../JwtAuthenticationGlobalFilter.java` 的 `isPublic`。

### 0.5 降级与运维总闸

- **客户端**：WS 不可用（握手失败、反复断链、服务端关闭等）时**自动退回**订单详情 **HTTP 轮询**，保证闭环。
- **服务端「开关」**：配置 **`passenger.ws.enabled`**（或与拒签 ws-token / 网关路由组合）作为**运维总闸**；关闭时全员无 WS，仅依赖轮询，便于故障止血（与客户端自动降级**并列**）。

### 0.6 多实例（后续）

- **已定选型 B**：**Redis Pub/Sub**（或等价 MQ）。发布侧仅投递 **`passengerId` + `orderNo` + `seq`** 等极小字段；**每台 `passenger-api`** 订阅，**仅本机存在该乘客 WS 会话时**向连接下行。
- **单机阶段**：可直接内存注册表推送；建议将「通知乘客」收口为 **`notifyPassenger(...)`**，单机直接发送、多机改为先 Pub 再本地发送，降低改造量。

### 0.7 乘客端弱网（客户端行为建议）

- **详情接口为最终权威**；收到推送后对 **GET 详情做短防抖**（如 300ms 内合并多次触发）。
- **有 WS 时**轮询间隔可**拉长**作兜底；**判定 WS 不可靠**后切换为**较短间隔轮询**，并对 **WS 重连使用指数退避 + 抖动**（封顶如 45s），避免重连风暴。
- **回前台 / 再次进入跟单页**：**先立即 GET 详情**，再尝试重建 WS；进程被杀则依赖下次进页。
- **总闸关闭**时：不反复尝试 WS，**静默纯轮询**。

## 1. 总体定位

| 维度 | 司机端（现状） | 乘客端（规划） |
|------|----------------|----------------|
| **服务对象** | 听单、待接指派、行程动作辅助 | 订单状态感知、重派中提示、终态通知等 |
| **核心实时诉求** | **指派列表变化**、后续可扩订单事件 | **当前行程/订单详情**与状态变化 |
| **BFF** | `driver-api`（8101，可直连或经网关） | `passenger-api`（8100，可直连或经网关） |
| **权威数据源** | 订单态仍以 `order-service` 为准；WS 为**展示/动效通道** | 同上：**HTTP 拉详情 / 写操作**仍为权威；WS **不替代**关键业务裁决 |

---

## 2. 连接与路径

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **Path 示例** | `/driver/ws/v1/stream` | 建议对称：`/app/ws/v1/stream`（经网关时加统一前缀，以网关规则为准） |
| **Query 鉴权** | `?token=<ws-token>`（握手用短期/专用 token） | 建议同样 **专用 ws-token**，避免把超长 `Authorization` 塞满 Query |
| **直连排障** | `VITE_DRIVER_WS_BASE_URL` 类环境变量绕过网关 | 乘客 H5 同理：`VITE_PASSENGER_WS_BASE_URL`（仅本地/dev） |
| **Spring 配置** | `@EnableWebSocket` + `WebSocketConfigurer` | 同构；若与 HTTP 同进程，注意 **Filter 放行 WS 握手路径**（司机端对 `/driver/ws` 有特殊 JWT `audit` 校验） |

---

## 3. 鉴权与令牌

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **HTTP 业务 Token** | `audit=1`（`DriverJwtService`） | 乘客 JWT：`aud=app-bff`（与现有 `passenger-api` 一致） |
| **WS 握手 Token** | `POST /driver/api/v1/auth/ws-token` → `audit=2`，**`tv` 与 HTTP 一致** | **已定**：`POST /app/api/v1/auth/ws-token`（路径以实现为准）→ **`audit=2`**，**必须 `tv`**，与顶号/登出一致 |
| **登出 / 顶号** | `tv` 递增 → 旧 JWT 失效；WS 应断开或握手失败 | **已定**：同司机；**禁止双端同时有效会话** |
| **Handshake Interceptor** | 校验 JWT、`tv`、审核状态等，session 写入 `driverId` | 校验乘客身份，session 写入 **`passengerId` / `customerId`（与 JWT sub 一致）** |

---

## 4. 会话模型

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **会话绑定键** | `driverId`（单终端策略：新连接踢旧连接以注册表为准） | **`passengerId`**；**已定**：**同一乘客仅一条 WS** + **`tv` 单会话**（与司机顶号语义对齐） |
| **订阅对象** | 默认等价「当前司机全部待接指派」 | **已定**：**跟单页建连**；**无进行中跟单则不长连**；多单并行本期不考虑 |
| **断线** | 可联动运力 **下线听单**（现状 WS 关闭会 `setOnline(false)`，与 HTTP 下线策略对齐） | 一般 **不建议**因 WS 断线自动取消订单；**最多标记「推送不可用」** |

---

## 5. 心跳与存活

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **应用层心跳** | 客户端 `ping`；服务端 `PONG` + 附带列表刷新（过渡策略） | 建议 **PING/PONG** 协议形态与**字段命名**与司机端一致，降低 H5 双端维护成本 |
| **心跳间隔** | 司机 H5 约 **15s** 发送 | 乘客端可 **15～30s**（弱网可略长）；**以服务端超时阈值为约束** |
| **服务端判死** | `heartbeat-timeout-ms`（如 45s）+ 定时扫 | 乘客端可 **更长**（乘客不派单、无 GEO 压力），避免杀连接过激进 |
| **业务含义** | 司机 WS 超时与 **听单 presence** 强相关 | 乘客 WS 超时 **一般不等价「取消行程」** |

---

## 6. 消息类型（现状 vs 建议）

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **已落地类型示例** | `ASSIGNED_LIST`、`ASSIGNED_ERROR`、`PONG` | 可分阶段：`ORDER_SNAPSHOT` / `ORDER_CHANGED` / 或 **「请刷新」型 `REFRESH_ORDER`** |
| ** payload 形态** | `type` + `ts` + `data`（JSON） | **建议同一套 envelope**：`type`、`ts`、可选 `messageId`、`data` |
| **推送内容** | 指派 **列表 VO**（过渡：服务端轮询 order 再推） | **已定 阶段 1**：**`orderNo` + 单调序号 `seq`**，客户端 **HTTP GET 详情**；**阶段 2** 再考虑嵌全量 `PassengerOrderDetailVO` |
| **错误可见性** | `ASSIGNED_ERROR` 便于联调 | 可对齐 `..._ERROR`，避免静默失败 |
| **去重** | 司机列表用 **hash** 去重推送 | **已定**：**`seq` 单调递增**去重/抗乱序（与 `messageId` 二选一亦可，团队对齐即可） |

## 7. 服务端推送从哪来

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **触发方式** | **定时拉 order** `listAssigned` + 哈希变更推送；接单/拒单后 **强制推送** | **阶段 1**：order 变更难以句句回调 passenger-api 时，可用 **「短轮询内部态 / 订阅 Outbox」** 或 **被动由关键 HTTP 写路径发 Redis Pub**；**阶段 2** 再收紧为事件驱动 |
| **与订单服务耦合** | Feign → `order-service` | 同；注意 **不要在 WS 线程里做重 IO**，异步发送 |
| **多实例** | 未上 Redis Pub 前，推送仅落连接所在节点 | **已定（乘客）**：**Redis Pub/Sub 广播 + 本机有会话再下行**（**B 方案**）；单机阶段内存即可 |

## 8. HTTP 与 WS 分工

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **读** | 可 **降级为纯 HTTP** 拉指派（`ENABLE_WS_ASSIGNED` 开关） | **已定**：**长期保留轮询兜底**；WS 失败**自动**退回轮询；另设 **服务端总闸** 关闭 WS（见 **§0.5**） |
| **写** | 接单/拒单/到达等 **均为 HTTP** | 取消订单、支付（若有）等 **一律 HTTP** |
| **产品感知** | 列表延迟 ≈ 推送周期 + 网络 | 详情延迟 ≈ 推送 + 一次 GET；需定义 **最大可接受延迟**（非硬实时） |

## 9. 网关与部署

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **WebSocket 支持** | 依赖网关开启 **Upgrade** 与路由 | **同**；乘客路由独立 `app/*` |
| **本地开发** | 常直连 `driver-api` 绕过网关未转发 WS | 乘客同理；**文档写明环境变量** |
| **线上** | WSS、域名证书、超时、连接数限制 | **同**；乘客连接数可能 **高于司机**（用户多、弱网重连多） |

---

## 10. 安全与合规

| 维度 | 司机端（现状） | 乘客端（规划建议） |
|------|----------------|-------------------|
| **Token 暴露** | ws-token 在 Query，需注意 **Referer/日志脱敏** | **同**；宜用 **短时效 ws-token** |
| **越权** | 会话绑定 `driverId`，业务接口再校验 | 会话绑定 `passengerId`，推送仅能是 **本人订单** |
| **payload** | 含订单位置等 | **最小化**；敏感字段与 HTTP 详情策略一致 |

## 11. 建议「优先对齐」清单（减少双端分裂）

1. **envelope**：`type` / `ts` / `data` / 可选 `messageId` 一致。  
2. **心跳**：`ping` / `PONG` 文本协议一致（或统一 JSON `{"type":"PING"}`）。  
3. **鉴权**：HTTP `audit=1` + WS `audit=2` + **ws-token 换取** 模式对称。  
4. **阶段 1 策略**：**WS 只负责「提醒刷新」**，业务数据 **仍以 HTTP 详情为准**（与司机端「列表过渡方案」同类）。  
5. **运维**：日志字段含 `userType` / `userId` / `orderNo`，便于一端问题不污染另一端排障。

## 12. 乘客端特有、不宜照搬司机端的点

| 项 | 说明 |
|----|------|
| **不与听单 GEO 耦合** | 乘客无「入池」语义；断线不应清 Redis 司机池。 |
| **进行中单 vs 无单** | **已定**：**无进行中跟单不长连**；跟单页建连、离页可断（见 **§0.3**）。 |
| **`reDispatching`** | 可由 **推送 + GET 详情** 结合；不必把复杂规则塞进 WS。 |
| **多端** | **已定**：**`tv` + 单 WS 连接**；新登录顶旧会话，**不允许多端同时有效 HTTP+WS**（与讨论结论一致）。 |
