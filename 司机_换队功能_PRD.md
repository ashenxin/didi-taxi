# 司机端换队（更换运力主体/车队）PRD

本文档定义 **司机端换队** 的产品闭环（提交申请、审核、撤销/放弃并恢复接单），并与后端边界对齐：

- **流程承载**：`capacity-service`
- **司机端对外入口**：`driver-api`（统一鉴权/统一返回；前端不直连 `capacity`）
- **管理后台入口**：`admin-api`（列表/详情/审核）

> 术语：本项目中“车队”不是独立资源表，一条 `company` 记录同时包含「公司 + 车队」信息；`teamId` 为车队业务编码（全库唯一）。

---

## 1. 目标与非目标

### 1.1 目标

- 司机可发起“换队”申请，选择目标运力主体（公司+车队）。
- 审核期司机 **不可接单**（与现有“先解绑”策略一致）。
- 管理后台可查看待审与历史申请，并审核通过/拒绝（原因可见）。
- 审核拒绝后司机可选择 **放弃换队并恢复接单**（方案 A），并且 **需要手动重新上线听单**。

### 1.2 非目标（本期不做）

- 自动推荐目标车队、复杂的跨城/跨省限制策略。
- 对外暴露 `capacity` 的 app 接口并直接让 H5 调用（避免鉴权与灰度复杂化）。
- 审核流程的多级审批、审批 SLA、消息推送等。

## 2. 核心业务规则

### 2.1 身份与安全

- 司机端所有业务接口均基于网关鉴权：
  - 请求头 `Authorization: Bearer <JWT>`
  - 网关注入 `X-User-Id = <JWT sub>`（字符串形式 `driverId`）
- 任何出现 `driverId` 的路径/Query/Body（若存在）必须与 `X-User-Id` 一致，否则 403。

### 2.2 申请地与一致性校验（口径）

- **申请地**：以司机档案 `driver.city_code` 为准。
- **一致性校验**：
  - 若 `driver.city_code != null`，则必须满足 `driver.city_code == fromCompany.city_code`；
  - 不满足时拒绝提交（提示“司机档案城市与当前归属运力城市不一致，请联系运营修正”）。
- **跨城换队**：允许司机选择任意目标城市/运力主体（不校验 `toCompany.city_code` 与申请地一致）。

### 2.3 状态机（申请单）

`driver_team_change_request.status`：

- `PENDING`：待审核
- `APPROVED`：通过
- `REJECTED`：拒绝
- `CANCELLED`：司机撤销/放弃（用于“恢复接单”的闭环）

状态流转：

- `PENDING` → `APPROVED`（管理后台审核通过）
- `PENDING` → `REJECTED`（管理后台审核拒绝，需原因）
- `PENDING` → `CANCELLED`（司机撤销申请并恢复接单）
- `REJECTED` → `CANCELLED`（司机“放弃换队并恢复接单”）

> 不支持：`APPROVED` → `CANCELLED`（已换队，不允许用“撤销”回滚）。

补充约束（产品口径）：

- **重复提交**：同一司机存在 `PENDING` 时不允许再次提交，提示“已有审核中申请”并引导去状态页。
- **并发互斥**：管理端审核与司机撤销/放弃同时发生时，仅允许一个成功（详见 TECH）。

### 2.4 司机接单资格（与现有实现对齐）

#### 2.4.1 提交申请后的副作用（先解绑）

提交成功后立即执行：

- `driver.company_id = null`（解绑当前归属）
- `driver.can_accept_order = 0`（不可接单）
- `driver.monitor_status` 不强制改动；但由于 `can_accept_order=0`，司机无法上线听单/接单

#### 2.4.2 审核通过

- `driver.company_id = toCompanyId`
- `driver.can_accept_order = 1`
- （实现策略）可同步 `driver.province_* / city_*` 为目标运力主体省市（以实现为准）

#### 2.4.3 审核拒绝

保持“解绑且不可接单”（不自动恢复）。

#### 2.4.4 司机放弃换队并恢复接单（方案 A）

司机在 `PENDING` 或 `REJECTED` 状态下可执行“放弃/撤销并恢复接单”：

- `request.status = CANCELLED`
- `driver.company_id = request.from_company_id`
- `driver.can_accept_order = 1`
- `driver.monitor_status = 0`（**必须手动重新上线** 才能听单）

## 3. 司机端页面与交互（H5/App）

### 3.1 入口与信息架构

- **换队入口**：工作台/我的/设置中增加“更换车队”入口。
- 页面拆分建议：
  1. **换队申请页**（目标搜索 + 原因 + 提交）
  2. **申请状态页**（当前/最新申请状态、审核结果、操作按钮）

### 3.2 换队申请页（提交）

#### 3.2.1 当前归属（只读展示）

展示当前司机归属信息（由后端按 `X-User-Id` 查询）：

- 当前城市（`driver.city_name`）
- 当前公司名称 + 车队名称（来自 `fromCompany`）
- 当前接单状态（可接单/不可接单、是否听单中）

#### 3.2.2 目标选择（3 个搜索框，模糊匹配 + 候选点选）

目标搜索条件（均可选，支持只填其中一个）：

- 城市（目标城市，可选）
- 公司关键字（可选）
- 车队关键字（可选）

搜索结果以列表展示（Top N、分页），每行至少包含：

- 目标城市 / 公司名称 / 车队名称

司机 **点选**一条，最终提交时使用 `toCompanyId`（不提交名称）。

#### 3.2.3 申请原因

- 文本可选（建议 0～200 字）

#### 3.2.4 强提醒（必须二次确认）

提交前弹窗强提醒：

- “提交后将 **暂停接单**，直到审核通过，或你选择 **放弃换队并恢复接单**。”
- “放弃换队恢复接单后，需要你 **重新上线听单**。”

### 3.3 申请状态页（查看/撤销/恢复）

展示“当前 `PENDING`（如有）否则最新一条申请”，并按状态展示操作：

- `PENDING`
  - 文案：审核中、暂停接单
  - 按钮：**撤销申请并恢复接单**
- `REJECTED`
  - 展示拒绝原因 `reviewReason`
  - 按钮：**放弃换队并恢复接单**
- `APPROVED`
  - 文案：已换队成功
  - 引导：可重新上线听单（若当前未听单）
- `CANCELLED`
  - 文案：已放弃换队并恢复接单
  - 引导：请重新上线听单

## 4. 管理后台流程（运营/审核）

管理后台按既有“换队审核”页面能力：

- 待审数角标
- 列表筛选（状态/司机手机号/时间等）
- 详情查看
- 审核通过/拒绝（拒绝原因必填）

运营侧 SOP（建议）：

- 对司机申请信息进行核对（司机档案、目标车队可用性等）
- 审核通过：司机归属切到目标车队、恢复可接单
- 审核拒绝：司机保持不可接单（由司机自助“放弃并恢复接单”或运营另行处理）

---

## 5. 关键体验与风控点（上线前必测）

- 重复提交：已有 `PENDING` 时提示并引导去“状态页”。
- 提交后的副作用：提交成功即不可接单/不可上线，必须明确告知。
- 撤销/恢复后：`monitor_status=0`，需要重新点击“上线听单”。
- 并发：在“审核通过”与“司机撤销”几乎同时发生时，需保证只有一个动作成功（CAS）。

---

## 6. 验收用例（最小集）

1. 司机提交申请 → 申请单 `PENDING` → 司机不可接单。
2. 管理端审核通过 → 申请单 `APPROVED` → 司机归属目标车队并可接单。
3. 管理端审核拒绝（含原因）→ 申请单 `REJECTED` → 司机仍不可接单 → 司机点击“放弃并恢复接单”→ `CANCELLED` → 司机可接单但需重新上线。
4. `PENDING` 阶段司机撤销 → `CANCELLED` → 恢复到原车队并可接单、需重新上线。
