# 后台管理系统：订单管理设计

本文档描述 **管理端订单域** 的产品边界、架构与数据权限要点，与《后台管理系统_订单管理_接口文档.md》配套。  
登录、角色、`sys_*` 模型见《后台管理系统_权限清单与鉴权设计.md》；接口与表契约不必重复。

---

## 1. 文档范围

| 包含 | 不包含 |
|------|--------|
| `didi-Vue` 订单列表/详情体验约定 | 网关统一鉴权 |
| `admin-api` 订单 BFF 职责、省/市数据域 | 司机端/乘客端下单流程 |
| 与 `order-service`、`passenger-service` 的依赖关系 | 人工改单、退款对账、导出（后置） |

---

## 2. 架构边界

- **`admin-api`**：对外前缀 **`/admin/api/v1/orders`**；校验 JWT；调用 **`OrderClient`**、**`PassengerClient`**；**不直连订单库**。
- **`order-service`**：`trip_order`、分页、详情、`order_event`。
- **`passenger-service`**：按手机号 / id 解析乘客，供列表筛选与详情补手机号。

---

## 3. 数据权限（与全站一致）

- 菜单仅控制「能否进入订单页」；**列表与详情**须遵守 **`AdminDataScope`**（见《后台管理系统_权限与接口文档.md》**§4.7**）。
- **非 SUPER**：列表查询的 `provinceCode`/`cityCode` 与 JWT 域合并；越界 **403**；详情订单若不在本账号省/市域内 **404**（文案「订单不存在」，不暴露它域是否存在订单）。
- **SUPER**：可按请求条件筛选，不做省/市锁定。

---

## 4. 业务能力约定

### 4.1 列表

- 筛选项：`orderNo`、`phone`、`provinceCode`、`cityCode`、`status`、创建时间起止、`pageNo`/`pageSize`（实现以接口文档为准）。
- **`orderNo` 与 `phone` 同时出现时**：有 `orderNo` 则按订单号为主；**仅**在无 `orderNo` 时用 `phone` → `passengerId` 再查订单。
- **按手机号筛选且乘客不存在**：**空列表**（业务 200），非报错。
- **列表不展示乘客手机号**：避免 N+1；与 BFF 对列表行清空 `passengerId`/`passengerPhone` 一致。若产品要强展示号码，需 passenger 批量 `by-ids` 或订单快照字段（后续）。

### 4.2 详情

- 返回订单主信息与 **事件时间线**（`occurredAt` 升序展示；缺失时可前端用 `createdAt` 兜底）。
- 有 `passengerId` 时 **一次** `GET /customers/{id}` 补 **详情内** 手机号。
- **排障增强**（主表全量时间节点、`cancelBy`、事件 `eventPayload` 折叠展示等）：非本期硬性范围，按迭代补充。

### 4.3 可选聚合

- 按 `driverId` 展示司乘公司/车辆等：依赖 **`capacity-service`**，当前以接口文档「是否已实现」为准。

---

## 5. 错误与体验

- **`ResponseVo.code` 与 HTTP 状态对齐**（401/403/404/502/504 等），前端对 `code !== 200` 统一用 `msg` 提示。
- 隐私脱敏、操作审计、导出：**首期不实施**，接口形状尽量预留扩展。
