本文档与《[网关服务_设计.md](网关服务_设计.md)》配套，供 **Spring Cloud Gateway** 模块 **编码与联调** 使用：依赖、配置键、路由样例、JWT/CORS/白名单/GatewayFilter 顺序及与现有 BFF 的对齐说明。

## 2. 工程结构

- 在仓库新增 Maven 子模块 **`gateway`**（`artifactId` 建议 `gateway`），`parent` 为 `com.sx:didi-taxi`。
- 包名建议：`com.sx.gateway`；主类 `@SpringBootApplication`，**不要**同时引入 `spring-boot-starter-web`（Servlet），网关基于 **WebFlux**。

**父 `pom.xml`**：在 `<modules>` 中增加 `<module>gateway</module>`。

**`gateway/pom.xml` 核心依赖**（版本由 BOM 管理，勿手写版本号 unless 无 BOM）：

| 依赖 | 作用 |
|------|------|
| `spring-cloud-starter-gateway` | 路由、过滤链、Netty |
| `spring-boot-starter-actuator` | `/actuator/health` 等 |
| `spring-boot-starter-security` | 与 JWT 验签/白名单组合（Reactive `SecurityWebFilterChain`） |
| `spring-boot-starter-oauth2-resource-server` | **可选**：若采用 Nimbus JWT 标准解码；与当前 jjwt 签发需核对算法与 claim |
| `jjwt-api` + `jjwt-impl` + `jjwt-jackson`（runtime） | **可选**：与 [admin-api JwtService](admin-api/src/main/java/com/sx/adminapi/auth/JwtService.java) **同库**验签，复用 HS256 逻辑 |

**推荐路径（落地快、与 admin-api 一致）**：网关中用 **JJWT** 在服务间约定密钥上验签，手写 **Reactive `GatewayFilter` 或 `WebFilter`** 解析 `Authorization: Bearer`，避免 OAuth2 Resource Server 与当前 token 形状不一致带来的调试。

## 3. 端口与下游 URI

| 实例 | 默认端口（本地） | 说明 |
|------|------------------|------|
| `gateway` | **8080**（建议，可配置） | 浏览器/H5 唯一对外 HTTP（开发期） |
| `admin-api` | 8099 | `uri: http://127.0.0.1:8099` |
| `passenger-api` | 8100 | `uri: http://127.0.0.1:8100` |
| `driver-api` | 8101 | `uri: http://127.0.0.1:8101` |

生产环境将 `uri` 换为 K8s Service 名或注册发现 `lb://admin-api`（若引入 **Spring Cloud LoadBalancer + 注册中心** 再改）。

### 3.1 WebSocket（规划，与司机端 / 乘客端实时通道对齐）

- **司机端**：浏览器/H5 经 **`gateway`** 与 **`driver-api`** 建立 **WebSocket**（`ws://` 本地 / `wss://` 生产），与 REST **同域同端口**；握手可用 **`Authorization: Bearer`** 或 **Query `token=`**（以小票换发方案为准，见《`司机端_上线听单与接单设计.md`》、司机 `DriverWsHandshakeInterceptor`）。
- **乘客端（已定稿）**：与司机**同形态** — 经网关 **`Path=/app/**`** 转发至 **`passenger-api`**（示例路径 **`/app/ws/v1/stream`**），与 《`乘客端与司机端_WebSocket_对比.md`》**§0 / §2** 对齐；**生产勿直连 BFF 端口**，本地排障可直连。
- **网关 JWT 与白名单（已落地）**：`JwtAuthenticationGlobalFilter` 对 **`GET /app/ws/`**、**`GET /driver/ws/`** 的 WebSocket **握手**放行（与 `/app/api` 的 Bearer 放行分离），由 BFF 从 **Query `token=`** 校验；**`GATEWAY_JWT_REQUIRE_AUTH=true` 时缺少本白名单会导致乘客 WS 握手 401**。代码：`gateway/src/main/java/com/sx/gateway/filter/JwtAuthenticationGlobalFilter.java` → `isPublic`。
- **网关能力**：须正确转发 **WebSocket 升级**（`Upgrade: websocket`），超时与 idle 配置需与长连接场景匹配（实现时调参）。浏览器 DevTools **Network** 选 **WS**，握手 **Status Code: 101 Switching Protocols**；**Fetch/XHR** 筛选项下**看不到** WS。
- **多实例**：连接落在某一实例。**乘客端已定**：**Redis Pub/Sub** 广播、`passenger-api` 各节点「有会话再下行」；司机端多副本时另可选 **Sticky** 或同等总线，避免推送丢失；详见《`司机端_上线听单与接单设计.md`》**§5.2**、《`乘客端与司机端_WebSocket_对比.md`》**§0.6**。

## 4. 路由配置（`spring.cloud.gateway`）

路由规则与设计文档 **§3** 对齐：**路径不重写**，整路径转发到 BFF。

**建议 `application.yml` 结构**（值为示例，可用环境变量覆盖 `GATEWAY_*`）：

```yaml
server:
  port: 8080

spring:
  application:
    name: gateway
  cloud:
    gateway:
      routes:
        - id: admin-bff
          uri: http://127.0.0.1:8099
          predicates:
            - Path=/admin/**
        - id: passenger-bff
          uri: http://127.0.0.1:8100
          predicates:
            - Path=/app/**
        - id: driver-bff
          uri: http://127.0.0.1:8101
          predicates:
            - Path=/driver/**
```

**注意**：

- 不要使用 `StripPrefix`，除非 BFF 侧 `context-path` 变更后统一调整。
- `Path` 区分大小写；与 Controller 前缀一致。

## 5. CORS（全局、唯一权威）

在 **`spring.cloud.gateway.globalcors`** 配置（与 [设计文档 §4](网关服务_设计.md) 一致），开发期可用 `allowedOriginPatterns`，生产改为明确列表。

**与 BFF 对齐**：三个 BFF 的 `CorsConfig` 已改为仅当 **`bff.browser-cors.enabled=true`** 时才注册；**并去掉 `admin-api` 各 Controller 上的 `@CrossOrigin(origins = "*")`**（否则会与网关 CORS 合并成非法的 `Origin, *`）。

**网关侧兜底**：模块内 `StripDownstreamCorsHttpHeadersFilter`（`HttpHeadersFilter` · RESPONSE）会在回写浏览器前**剔除下游带回的 `Access-Control-*`**，再仅由 `globalcors` 统一写出，避免旧实例或未拉代码的 BFF 仍带 `*` 时出现「双 Origin」。

**三 BFF 自检（与网关联调）**：

| 服务 | Controller `@CrossOrigin` | `CorsConfig` 默认 | `Security#cors` / 其它 |
|------|---------------------------|------------------|-------------------------|
| admin-api | 已清除（6 个 *Controller 均无） | 关闭，仅 `bff.browser-cors.enabled=true` 启用 | `SecurityConfig` 无 `.cors()` |
| passenger-api | 无 | 同上 | 无 Spring Security |
| driver-api | 无 | 同上 | 无 Spring Security |

**要点**：

- `allowedMethods` 含 `GET,POST,PUT,DELETE,PATCH,OPTIONS`。
- `allowedHeaders` 含 `Authorization,Content-Type` 及前端自定义头（按需补充）。
- **`OPTIONS`** 请求须返回 **204/200** 且带齐 CORS 头；Gateway 默认与 `globalcors` 配合即可，若叠加 **Spring Security**，需保证 **CORS 在 Security 链启用**（`ServerHttpSecurity.cors`），且 **OPTIONS 放行**（见 §7）。

**若使用 `allowCredentials: true`**：`allowedOrigins` 不可为 `*`，须枚举；并与前端约定。

## 6. JWT 与各端（admin/app/driver）对齐

### 6.1 当前签发（admin-api）

[JwtService#createToken](admin-api/src/main/java/com/sx/adminapi/auth/JwtService.java) 使用 **HS256**、`sub`=用户 id、`tv`=`token_version`、`uname`、**`iat`/`exp`**，并写入 **`aud=admin-bff`**（供网关校验）。

### 6.1.1 乘客端/司机端（签发约定）

- **乘客端（`/app/**`）**：`aud=app-bff`；签名密钥为 App 独立密钥（网关 `gateway.jwt.secret-app` 与乘客 BFF `app.jwt.secret` 对齐）。
- **司机端（`/driver/**`）**：`aud=driver-bff`；签名密钥为 Driver 独立密钥（网关 `gateway.jwt.secret-driver` 与司机 BFF `app.jwt.secret` 对齐）。
  - 司机端会在 BFF 侧校验 `tv`（token_version，Redis `driver:tv:{driverId}`）；网关不校验该细粒度逻辑。

### 6.2 网关验签（实现阶段）

| 步骤 | 说明 |
|------|------|
| 密钥（结构隔离） | **按路径前缀使用不同密钥**：`/admin/**` 用 **`JWT_SECRET_ADMIN`**（`gateway.jwt.secret-admin`），`/app/**` 用 **`JWT_SECRET_APP`**（`gateway.jwt.secret-app`），`/driver/**` 用 **`JWT_SECRET_DRIVER`**（`gateway.jwt.secret-driver`）。建议三端密钥彼此不同以实现跨端 token 的“验签级隔离”。 |
| 必验 | 签名、`exp` |
| 按路由验 `aud`（防御纵深） | 建议保持开启：网关按 `Path` 断言期望的 `aud`（`admin-bff` / `app-bff` / `driver-bff`）。即使有人误配了密钥，也能多一层拦截。 |
| 不验 | `tv`、菜单、省市区——仍在 BFF |

### 6.3 下行头（设计 §5.4）

- 校验通过后，设置 **`X-User-Id`** = JWT `sub`（字符串即可，与 BFF 约定）。
- **转发前删除**客户端传入的 `X-User-Id`（及任何拟信任的内部头），再注入网关解析值。

实现方式：**自定义 `GatewayFilter`**（顺序在路由元数据或 `Ordered` 接口中排在 strip/auth 之后），或使用 GlobalFilter；注意 **响应式线程** 中勿阻塞。

### 6.4 本地联调：整链关闭 JWT（仅开发）

`application.yml` 中 **`gateway.jwt.require-auth`** 默认 **`true`**；通过环境变量 **`GATEWAY_JWT_REQUIRE_AUTH=false`**（或 IDEA Run Configuration → *Environment variables*）可在本机 **跳过** 整段 Bearer 校验，便于 Postman / 无 token 前端联调。**生产环境必须为 `true`，切勿写入任何部署配置。**

关闭后：**不会**向 BFF 注入 `X-User-Id`；依赖该头的接口需在联调时自行带参或改用 token 模式。

## 7. 安全与白名单（Reactive Security）

使用 **`spring-boot-starter-security`** 时：

- 配置 **`SecurityWebFilterChain`**（不是 `HttpSecurity` Servlet）。
- **放行**（Public 白名单，登录/发码不带 JWT）：
  - 管理端：`POST /admin/api/v1/auth/login`
  - 乘客端（3 个）：
    - `POST /app/api/v1/auth/login-password`
    - `POST /app/api/v1/auth/login-sms`
    - `POST /app/api/v1/auth/sms/send`
  - 司机端（5 个）：
    - `POST /driver/api/v1/auth/sms/send`
    - `POST /driver/api/v1/auth/register-sms`
    - `POST /driver/api/v1/auth/register-password`
    - `POST /driver/api/v1/auth/login-sms`
    - `POST /driver/api/v1/auth/login-password`
  - 其它：`OPTIONS /**`、`/actuator/health`（及团队约定的其他 actuator；生产收紧）。
- **其余** `/admin/**`、`/app/**`、`/driver/**`：在 **`gateway.jwt.require-auth=true`**（默认）时要求 Bearer JWT（或由自定义 `GatewayFilter` 返回 401，Security 仅做 CSRF 关闭与 CORS）。见 **§6.4** 关闭校验的例外。

**顺序建议**：CORS → 认证（JWT）→ 路由转发。若 401/502 无 CORS 头，检查 **异常入口与 Spring Security 的 cors 融合**（见设计 §4）。

## 8. Actuator 与日志

- **暴露**：至少 `health`；`info`、`metrics` 按需；**勿**对公网暴露全量 actuator。
- **Request-Id**：可用 `spring-cloud-starter-gateway` 配合自定义 `GlobalFilter`，生成或透传 `X-Request-Id` / `traceparent`，与 BFF 现用 Micrometer/Zipkin 对齐方式在联调时统一命名。

## 10. 本地联调清单

1. 启动三个 BFF，确认 **直连** `8099/8100/8101` 可用。
2. 启动 `gateway`（8080）。
3. 前端/Vite：`baseURL` 指向 **`http://127.0.0.1:8080`**（或 `localhost:8080`，与 CORS 配置一致），路径仍为 `/admin/...`、`/app/...`、`/driver/...`。
4. **JWT**：正常路径下请求带 `Authorization: Bearer` 与有效 JWT；管理端可先 `POST http://127.0.0.1:8080/admin/api/v1/auth/login` 取 token。无 token 自测见 **§6.4**（`GATEWAY_JWT_REQUIRE_AUTH=false`）。

## 11. 开发任务拆分（建议顺序）

1. 新建模块、父 POM、启动类；拉通 **三条 Route**，无 JWT。
2. 配置 **globalcors**；用浏览器或 curl 验证 **OPTIONS** 与简单 GET。
3. 接入 **JWT 验签**（JJWT + 与 admin 同密钥）；白名单登录与 OPTIONS。
4. **剔除并注入 `X-User-Id`**；BFF 联调（可选：仅日志打印头）。
5. **签发侧增加 `aud`**（admin-api 先行），网关按路由校验 `aud`。
6. actuator、Request-Id、超时；限流熔断迭代。

## 13. 常见问题

**`NoSuchMethodError: HttpHeaders.headerSet()`**（栈在 `ForwardedHeadersFilter`）：Spring Cloud BOM **2023.0.5** 默认的 **Gateway 4.1.6** 需要 **Spring Framework 6.2**，而 **Spring Boot 3.3.5** 自带 **spring-web 6.1.x**，二者混用会报错。父工程 [pom.xml](pom.xml) 已通过 `dependencyManagement` 将 **`spring-cloud-starter-gateway`、`spring-cloud-gateway-server` 固定为 4.1.5**。日后若整体升级到 **Boot 3.4+** 与 **Spring Cloud 2024.0.x**，可删除该覆盖并随 BOM 升网关版本。

**macOS 上 Netty DNS 告警**（`MacOSDnsServerAddressStreamProvider` / `UnsatisfiedLinkError`）：`gateway/pom.xml` 已按 CPU 用 **profile** 引入 `io.netty:netty-resolver-dns-native-macos`（Apple Silicon：`osx-aarch_64`；Intel：`osx-x86_64`），与 Spring Boot 管理的 Netty 版本对齐。Linux/Windows 不激活上述 profile，无影响。
