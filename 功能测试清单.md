# didi-taxi 功能测试清单

本文档基于仓库后端模块与接口实现整理，用于**手工或自动化回归**时的测试说明。**实现以代码为准**；端口、URL 以各服务 `application.yml` 及部署环境为准（文中默认值为常见本地配置）。

| 元数据 | 说明 |
|--------|------|
| 适用范围 | `gateway`、`passenger-api`、`driver-api`、`order`、`capacity`、`calculate`、`map`、`passenger`、`admin-api` |
| 更新方式 | 功能变更时同步增删用例并改修订记录 |

---

## 一、文档约定

### 1.1 用例字段说明

每条用例建议至少包含以下信息（下表「测试细节」中已展开）：

| 字段 | 含义 |
|------|------|
| **用例 ID** | 唯一编号，便于缺陷关联 |
| **功能模块** | 所属服务或业务域 |
| **前置条件** | 数据、登录态、依赖服务 |
| **测试步骤** | 可执行的操作序列 |
| **预期结果** | 成功时的 HTTP 状态、业务码、库表/Redis 状态 |
| **备注** | 边界、负向、配置项 |

### 1.2 业务成功码约定

各 BFF 与领域服务若统一使用 `code=200` 表示业务成功，测试时以实际 `ResponseVo` / `CoreResponseVo` 为准。

### 1.3 典型本地端口（以默认配置为参考）

| 服务 | 默认端口 | 说明 |
|------|----------|------|
| gateway | 8080 | 统一入口：`/app/**`、`/driver/**`、`/admin/**` |
| passenger-api | 8100 | 乘客 BFF（网关转发） |
| driver-api | 8101 | 司机 BFF（网关转发） |
| admin-api | 8099 | 管理 BFF（网关转发） |
| order | 8093 | 订单服务 |
| capacity | 8090 | 运力服务 |
| calculate | 依项目配置 | 计费 |
| map | 依项目配置 | 地图路线 |
| Redis | 6379 | capacity 司机池 GEO、订单池 |
| MySQL | 3306 | 各业务库 |

## 二、环境与数据准备

### 2.1 ENV-01 基础环境可用

| 项 | 测试细节 |
|----|----------|
| **前置条件** | 无 |
| **测试步骤** | 1. 启动 MySQL、Redis。2. 启动各依赖微服务（至少 order、capacity；若测端到端则含 calculate、map、passenger-api、driver-api、gateway）。3. 访问各服务健康或 `/test/ping`（若存在）。 |
| **预期结果** | 服务无启动异常；数据库连接成功；Redis 可读写。 |
| **备注** | 无 Nacos 时按 `application.yml` 直连 URL 即可。 |

### 2.2 ENV-02 网关 JWT 策略

| 项 | 测试细节 |
|----|----------|
| **前置条件** | 启动 gateway |
| **测试步骤** | 1. 设置 `GATEWAY_JWT_REQUIRE_AUTH=false`，不带 Token 调用需登录接口，确认可联调。2. 生产或预发场景设置 `=true`，无 Token 应 401；带正确 `aud` 的 JWT 可访问对应端。 |
| **预期结果** | 与 `gateway.jwt` 配置一致；`X-User-Id` 仅由网关注入，客户端伪造头被剥离。 |
| **备注** | 参见 `JwtAuthenticationGlobalFilter`、`StripSpoofedUserHeaderGlobalFilter`。 |

### 2.3 ENV-03 测试账号与司机数据

| 项 | 测试细节 |
|----|----------|
| **前置条件** | capacity 库存在可接单司机：`can_accept_order=1`、`city_code` 非空（用于 GEO 删除与匹配）；必要时配置 `capacity.dispatch.geo-pin` 固定坐标联调。 |
| **测试步骤** | 1. 在 passenger 侧准备乘客账号与 `X-User-Id` 一致。2. 司机完成注册登录，取得 Bearer Token。 |
| **预期结果** | 下单、上线听单、指派链路可贯穿。 |
| **备注** | `city_code` 为空时，下线/登出可能无法从 GEO 移除（服务端会打 warn）。 |

| **预期结果** | 返回 token；`driver` 信息含 id、phone、audit 等。 |

### 5.2 DR-02 登出并清理司机池

| 项 | 测试细节 |
|----|----------|
| **前置条件** | 司机已登录听单（`online:true` 且 GEO 已写入） |
| **测试步骤** | 1. （可选）先造 **`ASSIGNED` / `PENDING_DRIVER_CONFIRM`** 指派给该司机。2. `POST /driver/api/v1/auth/logout`。3. 查 Redis `tx:driver:geo:{cityCode}` 中该成员不存在。4. 再请求听单接口应 401。5. （可选）查订单：待接指派应已 **`reject`**（`reasonCode=DRIVER_LOGOUT`），通常 **`CREATED`** 待重派，而非 **`CANCELLED`**。 |
| **预期结果** | 登出后 JWT 失效；运力侧 `monitor_status=0` 且 GEO 已删除（Feign `online:false`）；名下待接指派已从司机侧释放（批量拒单）。 |
| **备注** | 若 capacity 不可用，登出仍可能成功 token 侧，但需看 warn 日志。**`ACCEPTED`** 订单登出 **不会** 自动释放（与乘客登出代取消不同）。 |

### 5.3 DR-03 听单上线 / 下线

| 项 | 测试细节 |
|----|----------|
| **前置条件** | 司机 `can_accept_order=1` |

| **预期结果** | 上线后 DB `monitor_status=1`，GEO 有该司机；下线后为 0 且 GEO 删除。 |
| **备注** | 与登出删池逻辑一致。 |

### 5.5 DR-05 指派列表与详情

| **预期结果** | 后续变为 ASSIGNED/待确认。 |
